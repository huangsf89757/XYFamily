# 数据范围控制

> 权限校验必须结合数据范围（组织 / 团队 / 小组上下文）生效，确保账号只能操作其所属范围及下级范围内的资源，实现组织间完全的数据隔离。

## 二、业务规则

### 2.1 Scope 层级与上下文

- 数据范围层级：**组织 > 团队 > 小组**。
- 权限不是「全局生效」的，而是绑定在具体的范围上下文中，由请求携带其所处的范围标识。
- 全局操作（如 SuperAdmin 的系统配置、全局审计）无需范围上下文，由 `super_admin` 角色直接判定。

### 2.2 纵向覆盖（层级回溯）

- 高层范围的管理员权限自动覆盖其下属团队、小组范围，但仅覆盖**同层级**权限（详见 [权限继承](./04-权限继承.md) 的二维继承模型）。
- 通过成员关系回溯实现：当目标范围无成员记录时，向上查找（小组 → 团队 → 组织 → 全局）上级范围的角色，并以该上级角色作为生效角色参与校验。
- 示例：组织核心管理员（`organization_core_admin`）访问某团队接口时，即使其不在该团队的成员关系中，也可通过组织级成员关系回溯获得授权（因组织权限覆盖团队，且组织核心对应团队核心层级）。相反，组织普通管理员（`organization_ordinary_admin`）回溯到团队时仅具备团队普通管理员权限，不能执行团队结构性操作。

### 2.3 横向隔离（组织间隔离）

- **组织为租户边界**，组织间数据完全隔离。
- 同一账号在不同组织为独立成员关系，权限互不影响。
- 任何跨组织访问（未通过全局 SuperAdmin 角色）均被拒绝。

### 2.4 全局角色

- SuperAdmin（`super_admin`）游离于组织架构之外，不依赖任何成员关系即可跨组织执行操作，不受组织边界限制。

## 三、边界与异常处理

| 场景 | 处理方式 |
|------|----------|
| 范围上下文缺失 | 拒绝访问（接口要求范围但未提供） |
| 范围上下文非法（不属于该账号可访问范围） | 拒绝访问 |
| 跨组织越权访问 | 拒绝访问 |
| 账号在当前范围及上级范围均无成员关系 | 拒绝访问 |
| SuperAdmin 跨组织操作 | 允许（全局角色） |
