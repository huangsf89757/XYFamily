# 01-账号与认证域数据库设计

> P3 数据模型之一。定义账号、会话、邀请以及验证码 / Token 黑名单的存储结构。本域是认证模块（注册 / 登录 / 登出 / 刷新 / 重置）与账号模块（个人信息 / 生命周期）的落库依据，强依赖 P1 约束基线（C5 软删除、C6 安全常量）与 P2《JWT 鉴权链》（sessions / 黑名单结构）。

## 一、本域表 / 存储清单

| 存储 | 介质 | 说明 | PRD / ADR 依据 |
|------|------|------|----------------|
| `accounts` | PostgreSQL | 账号主表（聚合根） | ADR-005、ADR-006、FR-AUTH-001~008 |
| `sessions` | PostgreSQL | Refresh Token 有状态存储 | ADR-003、FR-AUTH-009/010 |
| `invitations` | PostgreSQL | 统一邀请流（ADR-009） | ADR-009、FR-ORG-004/TEAM-003/GROUP-004 |
| `verification_codes` | Redis | 验证码，单码单场景 | ADR-012、NFR-SEC-003、FR-AUTH-001/002/011 |
| `token_blacklist` | Redis | Access Token `jti` 黑名单 | ADR-003、FR-AUTH-010 |

> 软删除约定（ADR-006）：`accounts` / `*_members` **绝不硬删除**；认证域无 `*_members`，但 `invitations` 同样采用软删除（标记 `deleted_at`），仅 `accounts` 在注销后走匿名化（见 [05-支撑域/账号生命周期](../../05-支撑域/01-账号生命周期与身份方案.md)）。

## 二、`accounts`（账号主表）

```sql
CREATE TABLE accounts (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),   -- 数据库主键，对外隐藏
    account_id      UUID NOT NULL UNIQUE DEFAULT gen_random_uuid(), -- 业务主键，对外可见
    phone_encrypted   BYTEA,                                     -- 手机号密文（AES-256-GCM 加密，见 NFR-SEC-009）
    phone_hash        VARCHAR(64) UNIQUE,                        -- 手机号盲索引 HMAC-SHA256（用于等值查询）
    email_encrypted   BYTEA,                                     -- 邮箱密文（AES-256-GCM 加密）
    email_hash        VARCHAR(64) UNIQUE,                        -- 邮箱盲索引 HMAC-SHA256（用于等值查询）
    username          VARCHAR(64) UNIQUE,                         -- 用户名（不加密，非 PII）
    password_hash     VARCHAR(255),                               -- bcrypt cost=12，可空（第三方登录未绑定密码）
    nickname          VARCHAR(64),                                -- 昵称
    avatar            VARCHAR(512),                               -- 头像 URL（MinIO/S3）
    status            VARCHAR(16) NOT NULL DEFAULT 'active',      -- active|deactivating|deactivated
    deactivated_at    TIMESTAMPTZ,                                -- 注销宽限到期时间（宽限期满执行匿名化）
    last_login_at     TIMESTAMPTZ,                                -- 最后登录时间
    created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
    deleted_at        TIMESTAMPTZ                                 -- 软删除（实际不删除，仅匿名化）
);

-- PII 盲索引：登录等值查询走 phone_hash / email_hash（HMAC-SHA256 定长哈希，可 B-Tree 索引）
CREATE INDEX idx_accounts_phone_hash    ON accounts(phone_hash)  WHERE deleted_at IS NULL;
CREATE INDEX idx_accounts_email_hash    ON accounts(email_hash)  WHERE deleted_at IS NULL;
CREATE INDEX idx_accounts_username      ON accounts(username)    WHERE deleted_at IS NULL;
CREATE INDEX idx_accounts_status        ON accounts(status);
CREATE INDEX idx_accounts_deleted       ON accounts(deleted_at);
```

**字段说明**

| 字段 | 约束 / 取值 | 说明 |
|------|-------------|------|
| `id` | PK | 数据库内部主键，API 绝不暴露 |
| `account_id` | UNIQUE, NOT NULL | 业务聚合根标识，对外所有引用均用此值（ADR-005） |
| `phone_encrypted`/`email_encrypted` | AES-256-GCM 密文 | PII 加密存储（NFR-SEC-009）；密钥来自 KMS/环境变量 |
| `phone_hash`/`email_hash` | HMAC-SHA256 UNIQUE | 盲索引（Blind Index），用于登录等值查询；密钥独立于加密密钥 |
| `username` | UNIQUE | 用户名（非 PII，不加密）；全局唯一 |
| `password_hash` | bcrypt | cost=12（NFR-SEC-001）；第三方注册未设密码时为空 |
| `status` | `active` / `deactivating` / `deactivated` | 状态机见账号生命周期（FR-ACCT-004/005） |
| `deactivated_at` | - | 宽限期 30 天到期时间戳；到期由定时任务执行匿名化（ADR-010） |
| `deleted_at` | 软删除 | 注销后**不真删**，仅匿名化 PII（ADR-006/010） |

**约束**

- 唯一索引部分过滤 `WHERE deleted_at IS NULL`，复用同手机号/邮箱需先清原记录（注销场景）。
- 登录按 `phone_hash|email_hash|username|account_id` 自动识别类型走各自唯一索引（NFR 登录响应 <200ms）。

### PII 加密与盲索引方案（ARCH-001 修复）

| 维度 | 方案 |
|------|------|
| **加密算法** | AES-256-GCM（认证加密，带 nonce + tag），密钥来自 KMS / 环境变量 `XY_PII_ENCRYPTION_KEY` |
| **盲索引算法** | HMAC-SHA256，密钥独立于加密密钥，来自环境变量 `XY_PII_INDEX_KEY` |
| **存储** | `phone_encrypted`/`email_encrypted` 存密文（BYTEA），`phone_hash`/`email_hash` 存盲索引（VARCHAR(64)） |
| **写入流程** | 注册/修改手机号时：① 用加密密钥 AES-GCM 加密得到密文 → `phone_encrypted`；② 用索引密钥 HMAC-SHA256 得到哈希 → `phone_hash`；③ 两者同时写入 |
| **查询流程** | 登录时用 `phone_hash = HMAC(input_phone)` 做等值查询（命中 B-Tree 索引），查到后用解密密钥解密 `phone_encrypted` 验证 |
| **索引行为** | `phone_hash`/`email_hash` 为定长 hex 字符串，B-Tree 索引行为与普通 VARCHAR 一致，等值查询 O(log N) |
| **密钥轮换** | 加密密钥轮换时需全量重新加密；盲索引密钥轮换需全量重新计算 hash 并更新。建议低峰期执行，分批迁移 |
| **匿名化** | 注销匿名化时 `phone_encrypted`/`email_encrypted`/`phone_hash`/`email_hash` 全部置空或替换为 `anon_` 前缀 |

### 用户名修改保留期设计（MIN-009 修复）

> **背景**：PRD 审查 ADV-005 建议增加用户名修改保留期，防止用户修改用户名后原用户名被他人抢注造成混淆。

- **保留期**：用户名修改后，原用户名保留 30 天（配置 `account.username_reserve_days`），期间不可被其他用户注册。
- **实现方式**：accounts 表新增 `previous_username VARCHAR(64)` 和 `username_changed_at TIMESTAMPTZ` 字段。
- **校验逻辑**：注册/修改用户名时，除检查 `username` 唯一性外，还需检查全表 `previous_username` 是否匹配且 `username_changed_at + 保留期 > now()`。
- **保留期到期**：定时任务清理过期保留（`username_changed_at + 30d < now()` 时清空 `previous_username`），或随下次用户名修改覆盖。

```sql
ALTER TABLE accounts ADD COLUMN previous_username  VARCHAR(64);
ALTER TABLE accounts ADD COLUMN username_changed_at TIMESTAMPTZ;
CREATE INDEX idx_accounts_prev_username ON accounts(previous_username)
    WHERE previous_username IS NOT NULL;
```

## 三、`sessions`（Refresh Token 有状态存储）

```sql
CREATE TABLE sessions (
    id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    account_id        UUID NOT NULL,
    refresh_token_hash VARCHAR(255) NOT NULL,  -- 仅存哈希，不存明文
    device            VARCHAR(128),            -- 设备标识
    client_ip         INET,                    -- 客户端 IP（防重放/审计）
    user_agent        TEXT,                    -- UA
    expires_at        TIMESTAMPTZ NOT NULL,    -- Refresh 7d
    revoked_at        TIMESTAMPTZ,            -- 登出/重置/被盗用时置位
    created_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_sessions_account   ON sessions(account_id);
CREATE INDEX idx_sessions_expires   ON sessions(expires_at);
CREATE INDEX idx_sessions_revoked   ON sessions(revoked_at) WHERE revoked_at IS NULL;
```

**字段说明**

| 字段 | 说明 |
|------|------|
| `refresh_token_hash` | 仅存哈希（不可反解）；登录/刷新比对哈希 |
| `expires_at` | Refresh Token 7 天有效期（NFR-SEC-004） |
| `revoked_at` | 撤销时间：登出/改密/重置/被盗用触发（FR-AUTH-010、Rotation） |

**约束**：Refresh Token 一次性使用（One-Time Use），刷新成功后旧会话 `revoked_at` 置位并创建新会话（见 P2《JWT 鉴权链》§六）。

## 四、`invitations`（统一邀请流）

```sql
CREATE TABLE invitations (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    org_id              UUID NOT NULL,                 -- 所属组织（隔离边界）
    scope_type          VARCHAR(16) NOT NULL,          -- organization|team|group
    scope_id            UUID NOT NULL,                 -- 目标 scope 的 id
    inviter_id          UUID NOT NULL,                 -- 邀请人 account_id
    invitee_account_id  UUID,                         -- 已注册被邀请人（未注册为 NULL）
    invitee_contact     VARCHAR(255),                 -- 未注册时的 phone/email
    role                VARCHAR(32) NOT NULL,          -- 指定角色（必填，ADR-009）
    token               VARCHAR(255) NOT NULL UNIQUE,  -- 一次性邀请 Token
    status              VARCHAR(16) NOT NULL DEFAULT 'pending', -- pending|accepted|rejected|expired
    expired_at          TIMESTAMPTZ NOT NULL,          -- 7 天有效（ADR-009）
    created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
    accepted_at         TIMESTAMPTZ,
    deleted_at          TIMESTAMPTZ
);

CREATE INDEX idx_invitations_org       ON invitations(org_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_invitations_scope     ON invitations(scope_type, scope_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_invitations_invitee   ON invitations(invitee_account_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_invitations_contact   ON invitations(invitee_contact) WHERE deleted_at IS NULL;
CREATE INDEX idx_invitations_token     ON invitations(token);
```

**状态机**：`pending` → `accepted` / `rejected` / `expired`（见组织管理模块术语表「邀请状态机」）。Token 一次性使用，使用或过期后失效。

## 五、Redis：`verification_codes`（验证码，单码单场景）

依据 ADR-012，**单码单场景**：同一 `target` 可同时存在 register/login/reset_password 三个独立验证码，互不可串用。

| 项目 | 设计 |
|------|------|
| Key | `vc:{target}:{type}`（target=phone/email，type=register\|login\|reset_password） |
| Value | `{ code: "123456", attempt_count: 0, expires_at: <ts> }` |
| TTL | 5min（NFR-SEC-003） |
| 频控 | 同 target 1 次/60s；同 IP 5 次/5min（NFR-SEC-002/003） |
| 约束 | 消费时 `type` 必须匹配；`attempt_count >= 5` 或过期即失效 |

> 唯一性由 Key 设计天然保证（一 target 一 type 仅一码），等价于 PRD「唯一索引 `(target, type, status='pending')`」。

---

## 六、Redis：`token_blacklist`（Access Token 黑名单）

依据 ADR-003，无状态 Access Token 主动失效靠 Redis 黑名单。

| 项目 | 设计 |
|------|------|
| Key | `bl:{jti}`（jti 来自 Access Token Claims） |
| TTL | = Access Token 剩余有效期（避免永久存储） |
| 写入时机 | 登出（FR-AUTH-010）、修改密码（当前 Access）、密码重置（该账号全部会话） |
| 校验 | 每次请求在 JWTValidator 之后查黑名单，命中即 401（见 P4 中间件链） |

---

## 六-A、Redis：密码重置频率限制（DMC-007 修复）

> 满足 PRD FR-AUTH-011「同一账号 1 小时内最多 3 次密码重置」要求，认证接口错误码 `104292`（重置频控）。

| 项目 | 设计 |
|------|------|
| Key | `reset:acct:{account_id}:hourly`（account_id 为目标账号的 UUID） |
| Value | `{ count: 1, first_reset_at: <ts>, expires_at: <ts> }` |
| TTL | 3600s（1 小时滑动窗口） |
| 阈值 | `count >= 3` 即拒绝，返回 `104292`（重置频控超限） |
| 写入时机 | 密码重置请求成功时 `INCR` 并更新 TTL（首次设置 3600s，后续仅 INCR） |
| 约束 | 账号不存在时不写入频控（防枚举，FR-AUTH-011） |

> 密码重置后同时触发「该账号全部会话失效」：在 `token_blacklist` 中写入该账号所有活跃 `jti`，并在 `sessions` 表批量 `UPDATE ... SET revoked_at = now() WHERE account_id = ? AND revoked_at IS NULL`。
