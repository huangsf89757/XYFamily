## 二、严重问题（P0 — 必须修复）

### 🔴 ARCH-001：PII 加密存储与登录查询的技术矛盾未解决

| 项目 | 内容 |
|------|------|
| **位置** | `03-数据模型与契约/01-数据库设计/01-账号与认证域.md` §二 / `05-ER图与全局约定.md` §二 |
| **关联 PRD** | NFR-SEC-009（PII 加密存储） |
| **问题** | accounts 表注释标注 phone/email "加密存储见 NFR-SEC-009"，且建立了 `idx_accounts_phone ON accounts(phone) WHERE deleted_at IS NULL` 等索引。但若 phone 加密存储，等值查询（登录时 `WHERE phone = ?`）将无法命中 B-Tree 索引（加密后密文每次不同或不可比较）。架构文档未说明：① 加密算法选择（AES-GCM / AES-CBC / 哈希+盐）；② 加密后如何支持等值查询（是否采用盲索引 / hash 索引 / 应用层解密全扫描）；③ 索引在加密字段上的行为。 |
| **影响** | 登录性能 NFR-PERF-002（95% < 200ms）可能无法达成；如果采用应用层全解密扫描，在大用户量下不可行。这是影响系统可用性的核心架构问题。 |
| **建议** | 明确 PII 加密方案：推荐采用「盲索引（Blind Index）」方案——对 phone/email 计算 HMAC-SHA256 哈希作为独立查询列（如 `phone_hash`），建立索引；原始 phone/email 用 AES-GCM 加密存储。登录时用 `phone_hash` 查询，解密后验证。在数据库设计文档中补充此方案。 |

### 🔴 ARCH-002：权限缓存键粒度不足导致潜在越权风险

| 项目 | 内容 |
|------|------|
| **位置** | `02-核心域/02-RBAC权限引擎方案.md` §六 / `05-支撑域/02-缓存设计方案.md` §二 |
| **关联 PRD** | FR-PERM-002/003、NFR-SEC-012 |
| **问题** | 权限缓存键为 `perm:account:{account_id}:org:{org_id}`，值为"该账号在该组织下的有效权限点集合"。但用户在同一个 org 下的不同 team/group 可能有不同角色（如 team A 的 core_admin + team B 的 ordinary_admin）。缓存仅到 org 级别时，存储的是哪个角色的权限集？如果存储的是"最高角色权限集"（如 team_core_admin 的权限），则用户访问 team B 时可能获得超出其角色的权限（越权）。如果存储的是"按 scope 分别的权限集"，则缓存键不含 team/group 维度，无法区分。 |
| **影响** | 可能导致跨 scope 越权访问，违反 NFR-SEC-012（最小权限）。 |
| **建议** | 明确缓存策略：方案 A — 缓存键改为 `perm:account:{account_id}:{scope_type}:{scope_id}`（scope 含 org/team/group），精细化缓存；方案 B — 缓存键保持 org 级别，但存储的是 `{team_id: role, group_id: role}` 映射 + 各角色权限集，PermissionChecker 根据 MembershipValidator 确定的 effective_role 动态取对应权限集。推荐方案 A，虽然缓存条目增多但逻辑更清晰，命中率通过 scope 级别缓存仍可达标。 |

### 🔴 ARCH-003：审计日志 at-least-once 语义与仅追加约束冲突

| 项目 | 内容 |
|------|------|
| **位置** | `04-链路实现/03-审计日志方案.md` §一 / `03-数据模型与契约/01-数据库设计/04-审计域.md` §二 |
| **关联 PRD** | NFR-SEC-007（审计日志不可篡改） |
| **问题** | 审计方案使用 Redis Stream + consumer group 实现 at-least-once 投递，consumer 崩溃后从 `XPENDING` 恢复。但 `audit_logs` 表禁止 UPDATE/DELETE（仅追加）。当 consumer 在 `XACK` 之前崩溃并恢复时，会重新消费同一消息，导致重复 INSERT，产生重复审计记录。方案未说明去重机制（如事件 ID 幂等、唯一约束去重）。 |
| **影响** | 审计日志可能出现重复记录，影响审计准确性与合规性（NFR-COMPL-003）。 |
| **建议** | 增加审计事件唯一标识：① 在 Stream 消息中携带 `event_id`（如 UUID）；② `audit_logs` 表增加 `event_id UUID UNIQUE` 字段；③ consumer 插入前检查 `event_id` 是否已存在（或依赖唯一约束 INSERT ON CONFLICT DO NOTHING）。 |

### 🔴 ARCH-004：Token 黑名单与 Sessions 跨存储事务一致性缺失

| 项目 | 内容 |
|------|------|
| **位置** | `02-核心域/03-JWT鉴权链与Token方案.md` §五/七 / `03-数据模型与契约/01-数据库设计/01-账号与认证域.md` §三/六 |
| **关联 PRD** | FR-AUTH-010（登出）、FR-AUTH-011（密码重置） |
| **问题** | 登出操作需同时：① Redis 写入 `bl:{jti}`（黑名单）；② PG `UPDATE sessions SET revoked_at`。密码重置需同时：① Redis 写入多个 jti 黑名单；② PG 批量撤销 sessions。这两个操作跨 Redis 和 PG 两个存储，没有分布式事务保证。若 Redis 写成功但 PG 更新失败，Access Token 已失效但 Refresh Token 仍可用（用户可刷新获取新 Token，绕过登出）；若 PG 成功但 Redis 失败，Refresh 已撤销但 Access 仍可用（30min 窗口期内可操作）。 |
| **影响** | 登出/改密/重置的安全性存在时间窗口，可能被利用进行会话劫持。 |
| **建议** | 明确失败处理策略：① 优先写 PG（sessions 撤销是强一致操作），Redis 黑名单写入失败时记录补偿日志，由后台任务重试写入；② 或引入「先 PG 后 Redis」顺序，Redis 失败时返回成功（因为 PG 已撤销 Refresh，Access 最长 30min 自然过期），并告警。需在 JWT 方案中补充此异常处理流程。 |

## 三、重要问题（P1 — 强烈建议修复）

### 🟡 ARCH-007：PRD DEF-005 跨层级邀请成员关系创建逻辑未在架构中明确

| 项目 | 内容 |
|------|------|
| **位置** | `02-核心域/01-多租户隔离方案.md` / `03-数据模型与契约/02-接口设计/03-组织接口.md` §五 |
| **关联 PRD** | PRD 审查 DEF-005 |
| **问题** | PRD 审查 DEF-005 指出：用户通过小组邀请链接注册时，是否同时自动加入所属团队和组织？架构的多租户隔离方案和接口设计均未明确跨层级邀请的成员关系创建规则。若仅创建 group_members 而不创建 team_members/org_members，将出现"属于小组但不属于团队/组织"的悬空数据，导致层级回溯时该用户在 org/team 级别无成员关系，权限校验异常。 |
| **影响** | 跨层级邀请场景下数据一致性破坏，权限校验可能出现异常。 |
| **建议** | 在多租户隔离方案中补充跨层级邀请规则：小组邀请接受时，必须同时创建 `org_members`（role=regular_member）+ `team_members`（role=regular_member）+ `group_members`（role=指定角色），三者在同一事务中完成。团队邀请同理，需同时创建 `org_members` + `team_members`。 |
