## ADR 机制

### 目的

架构决策（Architecture Decision Record）用于沉淀"为什么这样做"的关键取舍，避免决策随人员变动而丢失，并让后续方案文档可追溯依据。

### 格式

每条 ADR 包含以下字段：

| 字段 | 说明 |
|------|------|
| 状态 | 提议中 / 已接受 / 已弃用 / 已取代（被取代时需注明取代的 ADR 编号） |
| 日期 | 决策接受日期 |
| 背景 | 面临的问题或约束（来自 PRD / NFR / 技术权衡） |
| 决策 | 最终选择的具体方案 |
| 理由 | 选择该方案的权衡依据 |
| 后果 | 接受后带来的正面收益与需承担的代价 / 后续约束 |

### 状态取值

- **提议中（Proposed）**：已提出但尚未评审接受。
- **已接受（Accepted）**：评审通过，作为实现依据。
- **已弃用（Deprecated）**：不再推荐，但历史上存在。
- **已取代（Superseded）**：被另一条 ADR 取代，注明 `取代 ADR-XXX`。

### 编号与维护

- 编号 `ADR-XXX` 顺序递增，不回收。
- 新增流程：提出决策 → 记录背景与备选方案 → 评审接受 → 在对应的 03 方案文档中落地。
- 当某条决策被推翻时，不删除原 ADR，而是将其标记为"已取代/已弃用"并新增一条，保留演进轨迹。

## ADR-001：超级管理员初始化方案

**状态**：已接受
**日期**：2026-07-12

**背景**：系统首次启动时数据库为空，没有用户。SuperAdmin 是唯一可以创建组织的角色，存在"鸡生蛋"问题。

**决策**：
- 生产环境：CLI `xyfamily admin init` 交互式创建，密码不进 shell history
- 开发/测试：环境变量 `XY_ADMIN_EMAIL` + `XY_ADMIN_PASSWORD` 快速初始化（只在数据库为空时执行一次）
- 初始密码 ≥12 位，创建后首次登录必须修改

**理由**：CLI 保证生产安全，环境变量保证开发效率。强密码 + 强制修改防止默认密码泄露。

**后果**：
- 首次部署流程增加一步（生产环境）
- 开发 CI 需配置初始化环境变量

## ADR-002：单角色 vs 多角色

**状态**：已接受
**日期**：2026-07-12

**背景**：PRD 设计了一个用户在组织中可有多个角色（如同时是 team_core_admin 和 group_core_admin），但六层角色体系（Public L0 + L1~L5）为单角色设计。

**决策**：采用单角色方案，每个 `*_members` 表含 `role` 字段，每个容器内一个账号只有一个角色。

**理由**：六层角色体系本质上是层级关系，不是多角色关系。一个账号在某组织内是 organization_core_admin，在某团队内是 RegularMember——角色是"在该容器中的位置"，不是"拥有的标签"。

**后果**：
- 不支持一人多角色场景（如一个团队内同时是 team_core_admin 和某小组的 group_core_admin）
- 如需此场景，需后续引入 `member_roles` 关联表

## ADR-003：Session 与 JWT 共存策略

**状态**：已接受
**日期**：2026-07-12

**背景**：PRD 同时定义 JWT（无状态）和 `sessions` 表（有状态），两者关系不清晰。

**决策**：
- Access Token：无状态 JWT，短有效期（30min），自然过期
- Refresh Token：有状态存储（`sessions` 表），登出时删除
- Token 黑名单：Redis（登出/改密/强制下线写入 jti，TTL = 剩余有效期）

**理由**：Access Token 无状态降低数据库压力；Refresh Token 有状态支持主动登出；Redis 黑名单兼顾安全与性能。

**后果**：
- 每次校验 Access Token 需查 Redis 黑名单（增加一次 Redis 调用，<1ms）
- 登出/改密必须写入黑名单，否则安全上有窗口期

## ADR-004：多租户隔离实现

**状态**：已接受
**日期**：2026-07-12

**背景**：PRD 要求"组织间完全隔离"，但未描述实现机制。

**决策**：应用层隔离（非 PG RLS），通过 `X-Organization-ID` / `X-Team-ID` / `X-Group-ID` Header 传递上下文，中间件强制校验 membership + 角色 + 权限点。

**理由**：应用层隔离更灵活，支持跨层权限继承（organization_core_admin 可操作团队而不需要先加入团队）。RLS 难以表达层级继承。

**后果**：
- 所有数据查询必须带 `org_id` 条件
- 权限中间件逻辑复杂（需实现层级回溯）
- 需要严格的代码审查防止隔离绕过

## ADR-005：账号唯一标识

**状态**：已接受
**日期**：2026-07-12

**背景**：PRD 支持手机号、邮箱、用户名、用户ID 四种登录方式，但未定义账号唯一标识体系。

**决策**：
- `account_id`（UUID）为业务主键，对外可见
- `id`（UUID）为数据库主键，对外隐藏
- phone/email/username 各自全局唯一
- 至少 phone 或 email 有一个非空

**理由**：双 UUID 设计保证数据库性能和 API 安全（不暴露自增 ID）。多登录标识支持灵活登录方式。

**后果**：
- 登录时需多字段查询（phone/email/username），索引设计需注意性能

## ADR-006：软删除策略

**状态**：已接受
**日期**：2026-07-12

**背景**：PRD 要求软删除，但未明确范围和约束。

**决策**：所有业务表使用 `deleted_at` 字段实现软删除，绝不对 `accounts` 和 `*_members` 表硬删除。

**理由**：audit 日志依赖 account_id 和 membership 引用，硬删除会破坏 referential integrity。

**后果**：
- 所有查询需排除 `deleted_at IS NULL`
- 匿名化时仅清空 role 字段，不删除 membership 记录

## ADR-007：角色解析——层级回溯 + 缓存预计算

**状态**：已接受
**日期**：2026-07-12

**背景**：一个账号可能同时是 Org A 的 organization_core_admin、Org B 的 RegularMember。权限校验需确定账号在当前 scope 的有效角色。

**决策**：
- `*_members.role` 是单一事实来源
- 权限中间件做层级回溯（account → group → team → org → global）
- Redis 缓存 `(account, org)` 有效权限集，key = `perm:account:{account_id}:org:{org_id}`，TTL 24h
- 角色/成员变更时主动失效缓存

**理由**：回溯保证正确性，缓存保证性能（p95 < 100ms）。两者不互斥。

**后果**：
- 权限中间件逻辑复杂（需实现回溯 + 缓存双重机制）
- 缓存失效需覆盖所有角色变更场景

## ADR-008：权限点矩阵——45 个 + Public 角色

**状态**：已接受
**日期**：2026-07-12

**背景**：PRD 定义 42 个权限点，但实际矩阵列出 45 个。`auth.register/login/reset_password` 无角色拥有。

**决策**：
- 45 个权限点（修正 PRD 的 42 旧数字）
- 新增 Public 角色（L0），每个用户自动拥有，拥有 3 个公开权限点
- Public 不与 membership 关联

**理由**：权限点矩阵逻辑纯度——所有权限点都有角色拥有者。中间件无需硬编码特例。

**后果**：
- 角色体系从 5 层变为 6 层（含 Public）
- 需在所有文档中统一更新角色层级表述

## ADR-009：成员邀请——统一邀请流

**状态**：已接受
**日期**：2026-07-12

**背景**：PRD 中邀请行为因用户是否存在而异（存在则直接加入，不存在则发链接），且角色可省略。

**决策**：无论被邀请人是否存在，均需对方确认后加入。邀请必须指定角色。创建 `invitations` 表统一管理。

**理由**：统一流程减少代码分支。角色必填让被邀请人明确责任。

**后果**：
- 所有邀请接口统一为 `POST /.../members/invitations`
- 增加"查看邀请→确认/拒绝"的用户流程
- 邀请有效期 7 天

## ADR-010：账号注销——30 天宽限期 + 匿名化

**状态**：已接受
**日期**：2026-07-12

**背景**：PRD 要求注销后 30 天宽限期，到期后处理机制缺失。

**决策**：
- 宽限期内保留成员身份，标记 `deactivating`，不可参与操作
- 到期后匿名化（不硬删除）：PII 替换为 `anon_` 前缀，membership 保留但 role 清空
- 绝不硬删除，保护 referential integrity

**理由**：审计日志依赖 account_id 和 membership 引用。硬删除会导致 audit 数据断裂。

**后果**：
- 定时任务每日扫描到期账号执行匿名化
- 匿名化不可逆（与宽限期内恢复区分）

## ADR-011：审计日志——双表 + Redis Stream 异步

**状态**：已接受
**日期**：2026-07-12

**背景**：审计日志必须不阻塞主流程，同时保证数据不丢失。

**决策**：
- `audit_logs` 单表（登录审计与操作审计同表，以 `action_type` 区分；原双表方案合并以简化），追加不可篡改
- Redis Stream + consumer group 异步写入，断点续写
- 操作类型枚举（登录类 + 组织/团队/小组/账号域操作类）

**理由**：Redis Stream 是既有依赖（验证码/限流/缓存已在用），支持 consumer group 保证不丢数据；单表便于按条件统一查询。

**后果**：
- 引入 Redis Stream 依赖
- 审计写入不再保证严格实时（毫秒级延迟）

## ADR-012：验证码——单码单场景

**状态**：已接受
**日期**：2026-07-12

**背景**：同一手机号可能被用于注册、登录、密码重置，存在跨场景串用风险。

**决策**：验证码生成时绑定 `type`（register/login/reset_password），消费时必须匹配。唯一索引 `(target, type, status='pending')`。

**理由**：唯一安全的选择。防止攻击者用一个场景的验证码消费另一个场景。

**后果**：
- 同一手机号同一时间可存在 3 个不同场景的验证码
- 发送频率：单手机号 1次/60秒，同 IP 5次/5分钟

## ADR-013：API 错误码体系

**状态**：已接受
**日期**：2026-07-12

**背景**：PRD 中 HTTP Status 和业务 Code 混用，错误码无规律。

**决策**：
- HTTP Status 表示请求层级（401/403/404/429/500）
- 业务 Code 表示具体错误，5 位数字分段：0=成功，10xxx=认证，20xxx=账号，30xxx=组织，40xxx=团队，50xxx=小组，60xxx=权限，70xxx=审计，80xxx=系统

**理由**：分层清晰，前端先看 HTTP Status 决定全局处理，再看业务 Code 决定局部提示。

**后果**：
- 所有响应格式需统一为 `{code, message, detail, data}`
- 创建接口 HTTP 201，其他 200

## ADR-014：组织/团队/小组创建——接棒机制

**状态**：已接受
**日期**：2026-07-12

**背景**：PRD 未说明创建组织/团队/小组时如何指定第一个管理员。

**决策**：
- 创建者自动成为对应层级的管理员（organization_core_admin/team_core_admin/group_core_admin）
- 可选通过 `initial_admin` 参数指定另一个人（走统一邀请流程）
- 最后一个管理员不可降级/移除/注销，必须先转让

**理由**：创建 + 指定管理员是同一业务意图，一步到位。最后管理员保护防止组织"无主"。

**后果**：
- 创建接口需支持 `initial_admin` 可选参数
- 降级/移除操作需检查是否为最后管理员

## ADR-015：第三方登录——延后

**状态**：已接受
**日期**：2026-07-12

**背景**：PRD 包含微信登录、第三方身份绑定等需求，但当前阶段聚焦核心认证能力。

**决策**：第三方登录（微信 OAuth2.0）、第三方身份绑定/解绑（FR-ACCT-006/007）延后到 P2。首期仅支持手机号/邮箱 + 密码/验证码登录。

**理由**：减少首期范围，聚焦核心 RBAC 底座。微信 OAuth2.0 涉及开放平台对接，独立开发更可控。

**后果**：
- `third_party_identities` 表暂不设计
- `verification_codes` 的 `type=bind` 场景延后
- 登录流程图中移除微信登录分支
