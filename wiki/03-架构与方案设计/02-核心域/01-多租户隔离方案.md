## 一、定位与 PRD 来源

PRD 要求"组织间完全数据隔离"（NFR-SEC-006），并以「组织 → 团队 → 小组 → 成员」四级结构承载企业级多租户数据隔离与权限治理（见 [多租户底座 PRD](../../02-需求与产品设计/01-产品PRD/01-多租户底座/多租户底座.md) 与 [组织管理模块 PRD](../../02-需求与产品设计/01-产品PRD/01-多租户底座/03-组织管理模块/组织管理模块.md)）。

本方案回答三个核心问题：

1. **租户如何建模**：三级租户 + 成员关系的层次结构。
2. **数据如何隔离**：应用层隔离（非 PG RLS），强制 `org_id` 贯穿所有查询。
3. **请求如何判定归属**：通过 `X-Org/Team/Group-ID` 上下文 + 成员关系回溯，确定请求生效的作用域与角色。

决策依据见 [ADR-004 多租户隔离实现](../01-基座/02-ADR架构决策记录.md)，约束基线见 [整体架构设计 C2/C4](../01-基座/01-整体架构设计.md)。

## 二、三级租户模型

```mermaid
graph TD
    ORG["Organization（组织）<br/>最高层级管理单元，数据隔离边界"]
    ORG --> TEAM["Team（团队）<br/>组织下属中层协作单元"]
    TEAM --> GROUP["Group（小组）<br/>团队下属基层执行单元"]
    GROUP --> MEM["Member（成员）<br/>归属于组织/团队/小组的账号"]
```

| 层级 | 实体 | 关键约束 | PRD 来源 |
|------|------|----------|----------|
| L3 组织 | `organizations` | 数据隔离边界；组织间完全隔离 | FR-ORG-001~009 |
| L2 团队 | `teams` | 隶属于某组织；含 `archived_at` 软归档 | FR-TEAM-001~009 |
| L1 小组 | `groups` | 隶属于某团队；含 `deleted_at` 软删除 | FR-GROUP-001~008 |
| 成员 | `*_members` | 复合唯一 `(scope_id, account_id)`；单角色列 `role` | FR-ORG/TEAM/GROUP-007 |

**关键语义**
- 一个组织下可有多个团队，一个团队下可有多个小组；所有团队/小组最终归属于唯一组织。
- 组织是租户边界，**组织间完全数据隔离**；同一账号在不同组织为相互独立的成员关系，权限互不影响。
- 父子关系由外键承载：`teams.org_id` → `organizations`，`groups.team_id` → `teams`（并冗余 `org_id` 便于索引与隔离校验）。

## 三、数据隔离策略

### 3.1 应用层隔离（非 RLS）

采用 **应用层隔离**（ADR-004），理由：

- 权限继承需要跨层表达（组织核心管理员可操作团队而不必先加入团队），RLS 难以表达层级回溯。
- 隔离逻辑集中在中间件，便于统一审计与测试。

**强制约束（C2）**
- 所有业务表携带 `org_id`（组织 id），作为隔离主键。
- 所有数据查询必须带 `org_id` 条件，缺失即拒绝（防止越权读取）。
- 跨组织访问（未通过全局 SuperAdmin）一律拒绝，返回 403。

### 3.2 软删除与隔离的共存

- 组织/团队/小组均使用 `deleted_at` / `archived_at` 软删除（ADR-006）；隔离查询需同时排除已删除行。
- 软删除不破坏父子外键与成员引用，保证审计完整性。

### 3.3 索引与性能

- 所有租户相关表 `org_id` 必带索引；`(scope_id, account_id)` 复合唯一索引保证成员唯一。
- 隔离过滤走索引前缀，避免全表扫描（NFR-PERF-001：95% < 100ms）。

## 四、租户上下文传递

每个需鉴权的请求通过 Header 显式携带其作用域上下文：

| Header | 含义 | 必填 |
|--------|------|------|
| `X-Organization-ID` | 目标组织 id | 组织/团队/小组域接口必填 |
| `X-Team-ID` | 目标团队 id | 团队/小组域接口必填 |
| `X-Group-ID` | 目标小组 id | 小组域接口必填 |

**推断规则**
- 单组织用户（仅属一个组织）可省略 `X-Organization-ID`，后端依据 Token Claims 中的 `org_ids` 自动推断。
- 多组织用户（≤10 个，见 Token Claims）必须显式指定，否则返回 400（上下文缺失）。
- 父子 scope 必须自洽：`X-Group-ID` 所属团队必须等于 `X-Team-ID`，其所属组织必须等于 `X-Organization-ID`；不一致返回 400。

---

## 五、上下文校验与层级回溯

权限校验前，必须先确定"当前账号在该 scope 的生效角色"。当目标 scope 无直接成员记录时，向上回溯上级 scope 的角色（见 [RBAC 权限引擎方案](./02-RBAC权限引擎方案.md) 的二维继承与 [数据范围控制 PRD](../../02-需求与产品设计/01-产品PRD/01-多租户底座/06-权限管理模块/03-数据范围控制.md)）。

```mermaid
flowchart TD
    A[请求携带 X-Org/Team/Group-ID] --> B{上下文自洽?}
    B -->|否| B1[返回 400]
    B -->|是| C{目标 scope 有直接成员?}
    C -->|是| D[使用该 scope 角色]
    C -->|否| E[向上回溯: group→team→org→global]
    E --> F{回溯到某上级成员?}
    F -->|是| G[以该上级角色作为生效角色<br/>（按二维继承取同层级权限）]
    F -->|否| H[返回 403 非成员]
    G --> I[进入权限点校验]
    H --> I
```

**回溯示例**
- `organization_core_admin` 访问团队接口：虽不在该团队的 `team_members`，但凭组织级成员关系回溯获得授权（组织权限覆盖团队，对应团队核心层级）。
- `organization_ordinary_admin` 回溯到团队：仅具备团队普通管理员权限，不能执行团队结构性操作（纵向仅继承同层级）。

**全局角色例外**
- `super_admin` 不依赖任何成员关系，可跨组织执行操作，不受组织边界限制（ADR/PRD 角色体系 L8）。

## 五-A、跨层级邀请成员关系创建规则（ARCH-007 修复）

> **背景**：PRD 审查 DEF-005 指出，用户通过小组/团队邀请链接注册时，是否同时自动加入上级层级未明确。若仅创建 `group_members` 而不创建 `team_members`/`org_members`，将出现「属于小组但不属于团队/组织」的悬空数据，导致层级回溯时权限校验异常。

### 规则定义

| 邀请层级 | 接受时创建的成员关系 | 事务要求 |
|----------|----------------------|----------|
| 小组邀请 | `org_members`（role=regular_member）+ `team_members`（role=regular_member）+ `group_members`（role=指定角色） | 三者在同一 PG 事务中完成 |
| 团队邀请 | `org_members`（role=regular_member）+ `team_members`（role=指定角色） | 两者在同一 PG 事务中完成 |
| 组织邀请 | `org_members`（role=指定角色） | 单条 |

### 实现要点

1. **幂等处理**：邀请接受前先检查 `org_members`/`team_members` 是否已存在该 `account_id`，已存在则跳过创建（用户可能已在上级层级）。
2. **角色分离**：上级层级自动创建的成员关系统一为 `regular_member`，不继承邀请指定的角色（邀请角色仅作用于目标 scope）。
3. **缓存失效**：成员关系创建后，需失效该账号在相关 scope 的权限缓存（`perm:account:{id}:org:{org_id}` 等，见 [缓存设计方案](../05-支撑域/02-缓存设计方案.md)）。
4. **审计**：每条成员关系的创建均记录审计日志（`action_type` = `org.member.invite` / `team.member.invite` / `group.member.invite`）。

### 时序示例（小组邀请接受）

```mermaid
sequenceDiagram
    participant A as 被邀请人
    participant API as API
    participant PG as PG

    A->>API: POST /invitations/{token}/accept
    API->>PG: BEGIN TRANSACTION
    API->>PG: 校验 invitation token（一次性 + 未过期）
    API->>PG: INSERT org_members (account_id, role=regular_member) ON CONFLICT DO NOTHING
    API->>PG: INSERT team_members (account_id, role=regular_member) ON CONFLICT DO NOTHING
    API->>PG: INSERT group_members (account_id, role=指定角色)
    API->>PG: UPDATE invitations SET status=accepted
    API->>PG: COMMIT
    API->>API: 失效权限缓存
    API-->>A: 200 加入成功
```

## 五-B、组织归档/停用处置策略（ARCH-013 修复）

> **背景**：PRD 审查 ADV-003 指出组织层面缺少归档/删除功能。组织是租户根节点，硬删除影响过大（级联团队/小组/成员/审计），因此采用「停用」策略替代硬删除。

### 处置策略

| 操作 | 接口 | 权限 | 效果 |
|------|------|------|------|
| 停用组织 | `POST /api/v1/organizations/{id}/disable` | 仅 SuperAdmin | 组织标记 `disabled_at`，所有成员角色权限暂停，数据保留 |
| 启用组织 | `POST /api/v1/organizations/{id}/enable` | 仅 SuperAdmin | 清除 `disabled_at`，成员角色权限恢复 |

### 级联处理规则

1. **组织停用时**：
   - 组织下所有团队/小组标记为「继承停用状态」（不单独设字段，查询时联查组织状态）
   - 所有成员的权限校验立即拒绝（中间件检查组织 `disabled_at`，非 NULL 则返回 `403`）
   - 用户登录仍可成功（账号级操作不受影响），但无法执行组织内业务操作
   - 审计日志正常记录

2. **组织启用时**：
   - 清除 `disabled_at`
   - 成员权限自动恢复（无需重新分配）
   - 权限缓存需批量失效

3. **不硬删除**：组织数据永久保留，满足审计可追溯性（NFR-SEC-007）；企业退订场景通过停用实现，PII 数据在账号注销时匿名化。

### 数据库变更

`organizations` 表新增 `disabled_at TIMESTAMPTZ` 字段（标记停用时间，NULL 表示正常）。
