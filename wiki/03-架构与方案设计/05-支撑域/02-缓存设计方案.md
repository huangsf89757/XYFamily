# 02-缓存设计方案

> P5 支撑域之二。定义 Redis 在全系统的缓存使用全景：验证码、Token 黑名单、权限缓存、登录限流、热点数据。承接 [P2 RBAC §六（权限缓存）](../../02-核心域/02-RBAC权限引擎方案.md) 与 [P2 JWT §五（黑名单）](../../02-核心域/03-JWT鉴权链与Token方案.md)，满足 NFR-PERF-008（命中率 ≥95%）、NFR-PERF-001（95% < 100ms）。

## 一、缓存键全景

| 用途 | Key | 类型/值 | TTL | 依据 |
|------|-----|---------|-----|------|
| 验证码 | `vc:{target}:{type}` | String/Hash `{code, attempt_count, expires_at}` | 5min | ADR-012、NFR-SEC-003 |
| 验证码频控(账号) | `vc:acct:{account_id}:daily` | String（计数） | 24h | ARCH-011 修复、NFR-SEC-003 |
| Token 黑名单 | `bl:{jti}` | String（空值） | = Access 剩余有效期 | ADR-003 |
| 权限缓存 | `perm:account:{account_id}:{scope_type}:{scope_id}` | Set（权限点集合） | 24h | ADR-007、P2 RBAC §六、ARCH-002 修复 |
| 登录限流(IP) | `rl:ip:{ip}` | String（计数） | 300s 窗口 | NFR-SEC-002 |
| 登录限流(账号) | `rl:acct:{account_id}` | String（计数） | 300s 窗口 | NFR-SEC-002 |
| 热点组织信息 | `org:info:{org_id}` | Hash（name/desc/owner） | 300s | NFR-PERF-008 |
| 审计 Stream | `audit_stream` | Stream | 持久 | ADR-011 |

## 二、权限缓存（核心）

> **ARCH-002 修复**：缓存键从 org 级别精细化到 scope 级别。

- **读取路径**（[P4 中间件 PermissionChecker](../04-链路实现/01-中间件链专项方案.md)）：MembershipValidator 先确定请求的生效 scope（org/team/group），然后查 `perm:account:{id}:{scope_type}:{scope_id}`；未命中查 `role_permissions` 预填充展开并回填。
- **失效时机**（主动删除 key）：
  - 角色/成员关系变更（分配/降级/移除/邀请接受）——需删除该账号在该 scope 及其下级 scope 的所有缓存 key
  - 权限矩阵重新预填充（[P2 RBAC §五](../../02-核心域/02-RBAC权限引擎方案.md)）——需清空所有 `perm:account:*` key
- **降级**：Redis 不可用时回退 DB 查询，不阻断业务（仅性能下降）。
- **命中率目标 ≥95%**（NFR-PERF-008）。

---

## 三、Token 黑名单

- 写入：登出（当前 Access jti）、改密（当前 Access jti）、重置（该账号全部会话相关 jti）。
- TTL 设为 Access 剩余有效期，避免永久存储、自然过期。
- 每次请求在 JWTValidator 之后查一次（<1ms）。

---

## 四、登录限流

- 双维度：`rl:ip:{ip}`（5min 5 次 → 锁 15min）+ `rl:acct:{account_id}` 叠加。
- 命中返回 `429`（104290/104291）。
- 使用原子递增 + `EXPIRE` 窗口。

## 四-A、验证码账号级频控（ARCH-011 修复）

- **背景**：PRD 审查 DEF-016 指出验证码发送限流缺少账号级别限制，攻击者可通过分布式 IP 对特定用户进行短信轰炸。
- **频控键**：`vc:acct:{account_id}:daily`，TTL 24h，阈值 20 次/天。
- **检查时机**：发送验证码时，除 target 1次/60s 和 IP 5次/5min 外，额外检查账号级日频控。
- **命中返回**：`429`（114293 — 账号级验证码频控超限）。
- **未登录场景**：若账号不存在（如注册场景），仅依赖 target + IP 频控；密码重置场景按 target 反查 account_id 后施加账号级频控。

---

## 五、热点数据

- 组织信息、角色权限等读多写少数据缓存（NFR-PERF-008）。
- 写操作（更新组织/角色）后失效对应缓存。

---

## 六、高可用与运维

- Redis Sentinel 保证高可用（[P1 部署架构](../../01-基座/01-整体架构设计.md)）。
- 缓存雪崩防护：TTL 加随机抖动；穿透防护：空值短缓存；击穿防护：单 key 重建加分布式锁。
- 监控：命中率、内存、连接数（[P6 可观测性](../../06-横切专项/03-可观测性方案.md)）。
