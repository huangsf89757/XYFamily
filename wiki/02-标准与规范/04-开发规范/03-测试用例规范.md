## 一、用例编号规则与命名规范

### 1.1 编号格式

TC-{产品线}-{模块}-{层级}-{序号}

| 组成 | 说明 | 示例 |
|------|------|------|
| TC | Test Case 前缀，固定值 | TC |
| {产品线} | 产品线缩写，大写英文 | XYF |
| {模块} | 功能模块缩写，大写英文 | AUTH、USER |
| {层级} | 测试层级缩写（见 1.3） | UNIT、API、E2E |
| {序号} | 四位流水号，从 0001 起 | 0001 |

完整示例：TC-XYF-AUTH-API-0001 表示「XYF 产品线-认证模块-接口测试-第 0001 条用例」

> 模块缩写由各项目自行定义，在项目测试计划中维护对照表。编号一经分配不可复用，废弃用例保留编号占位。

### 1.2 用例名称规范

- 格式：{测试意图} — {场景/条件}
- 示例：手机号登录 — 密码正确、创建资源 — 名称重复、分页查询 — 超出总页数
- 语气：陈述句，避免疑问句
- 长度：不超过 30 个中文字符
- 禁止使用「测试 xxx」前缀，直接描述测试意图

### 1.3 测试层级缩写

| 层级 | 缩写 | 说明 |
|------|------|------|
| 单元测试 | UNIT | 函数/方法级别，Mock 外部依赖 |
| 接口测试 | API | HTTP/RPC 接口级别，真实网络请求 |
| 集成测试 | INT | 多模块联合，跨 Service 边界 |
| 端到端测试 | E2E | 端到端，模拟真实用户操作链路 |

## 二、优先级定义（P0~P3）

| 优先级 | 含义 | 覆盖要求 | CI 阻断策略 |
|--------|------|----------|------------|
| P0 | 核心主流程，阻塞发布 | 100% 覆盖 | 失败阻断合入 |
| P1 | 重要功能路径，发布前必须通过 | 不低于 95% | 失败阻断发布，需产品确认方可豁免 |
| P2 | 边界与异常场景 | 不低于 80% | 失败记录跟踪，不阻断合入 |
| P3 | 极端边界与 UI 细节 | 按需覆盖 | 不阻断 |

### 优先级判定原则

- P0：核心业务主路径、资损/安全相关、数据一致性保障
- P1：重要辅助功能路径、高频使用场景
- P2：边界值、异常输入、并发/竞态场景
- P3：UI 交互细节、极端数据、兼容性边缘

## 三、用例设计方法

> 遵循经典软件测试设计方法，确保用例覆盖全面、无冗余。

### 3.1 等价类划分法

将输入域划分为有效等价类和无效等价类，每类取代表性值。

| 类型 | 示例 |
|------|------|
| 有效等价类 | 密码长度 8~32 位，含大小写+数字 |
| 无效等价类 | 密码 < 8 位 / > 32 位 / 纯数字 / 纯字母 |

### 3.2 边界值分析法

针对边界及其邻近值设计用例，覆盖 off-by-one 错误。

| 边界 | 测试点 |
|------|--------|
| 最小值 min | min-1, min, min+1 |
| 最大值 max | max-1, max, max+1 |
| 分页 size | 0, 1, 额定上限, 上限+1 |

### 3.3 判定表法

多条件组合场景，构建判定表，每列对应一条用例。

| 条件1 | 条件2 | 预期动作 |
|-------|-------|---------|
| Y | Y | 动作 A |
| Y | N | 动作 B |
| N | Y | 动作 C |
| N | N | 动作 D |

### 3.4 状态迁移法

适用于有状态机的业务（如订单状态、账号生命周期），覆盖所有合法迁移路径及非法迁移尝试。

### 3.5 错误推测法

基于经验推测可能出错的场景，补充到用例库中。

- 空值 / null / undefined
- 超长字符串 / 特殊字符 / SQL 注入
- 并发操作 / 竞态条件
- 网络超时 / 断连重连
- 时区 / 跨日期操作

### 3.6 正交实验法

多因素组合测试，使用正交表减少用例数量同时保证覆盖度。适用于配置组合、环境矩阵等场景。

## 四、用例字段说明

### 4.1 必填字段

| 字段 | 类型 | 说明 |
|------|------|------|
| 用例编号 | string | 遵循编号规则，全局唯一，不可复用 |
| 用例名称 | string | 遵循命名规范 |
| 优先级 | P0~P3 | 优先级定义 |
| 测试层级 | UNIT/API/INT/E2E | 测试层级缩写 |
| 前置条件 | text | 执行用例前必须满足的条件（数据状态、环境配置、用户角色等） |
| 测试步骤 | text[] | 有序的操作步骤，每步一行，可被新人复现 |
| 预期结果 | text | 步骤执行后应观察到的正确行为/数据状态，必须可断言 |
| 关联需求 | string | 需求编号（如 FR-XXX-001） |

### 4.2 可选字段

| 字段 | 类型 | 说明 |
|------|------|------|
| 关联接口 | string | API 路径（如 POST /api/v1/xxx） |
| 测试数据 | text | 构造测试数据的脚本、SQL 或 fixture |
| 后置清理 | text | 用例执行后的数据清理操作 |
| 设计方法 | string | 等价类/边界值/判定表/状态迁移/错误推测/正交 |
| 备注 | text | 额外说明 |

### 4.3 执行结果字段（运行时填写）

| 字段 | 类型 | 说明 |
|------|------|------|
| 实际结果 | text | 实际观察到的行为 |
| 执行状态 | pass / fail / blocked / skip | 用例执行结果 |
| 执行人 | string | 执行者标识 |
| 执行时间 | datetime | 执行时间戳 |
| 关联 Bug | string | 失败时关联的 Bug ID |
| 失败原因 | text | 失败时填写根因分析 |

## 五、用例编写模板

### 5.1 接口测试用例模板

| 字段 | 内容 |
|------|------|
| 用例编号 | TC-{产品线}-{模块}-API-{NNNN} |
| 优先级 | ../../P3 |
| 关联接口 | {METHOD} {PATH} |
| 前置条件 | 1. ... 2. ... |
| 测试步骤 | 1. 发送 {METHOD} {PATH}，请求体: {...} |
| 预期结果 | 1. HTTP 状态码 = {expected_status} 2. 响应业务码 = {expected_code} 3. 响应 data 字段值正确 |
| 后置清理 | {cleanup_action} |

### 5.2 单元测试用例模板

| 字段 | 内容 |
|------|------|
| 用例编号 | TC-{产品线}-{模块}-UNIT-{NNNN} |
| 优先级 | ../../P3 |
| 前置条件 | Mock {dependency} 返回 {value} |
| 测试步骤 | 1. 调用 {function}({params}) |
| 预期结果 | 1. 返回值 = {expected_value} 2. {dependency}.{method} 被调用 {n} 次 |
| 后置清理 | 无（Mock 自动清理） |

### 5.3 端到端测试用例模板

| 字段 | 内容 |
|------|------|
| 用例编号 | TC-{产品线}-{模块}-E2E-{NNNN} |
| 优先级 | ../../P3 |
| 前置条件 | 1. 测试环境已启动 2. 测试账号已创建 3. 依赖数据已就绪 |
| 测试步骤 | 1. 打开页面 A 2. 输入凭证，点击操作按钮 3. 等待页面跳转/弹窗/接口响应 |
| 预期结果 | 1. 页面跳转至预期路径 2. 页面元素显示预期内容 |
| 后置清理 | 删除测试数据，恢复初始状态 |

## 六、用例评审流程

评审流程：用例编写人提交 -> 技术负责人初审 -> 产品/需求方复审 -> 合入用例库 -> 关联需求编号 -> 更新覆盖率统计

### 评审检查清单

| 检查项 | 要求 |
|--------|------|
| 编号唯一性 | 用例编号在用例库中无重复 |
| 需求覆盖 | 每个 P0 需求至少有 1 条 P0 用例 |
| 前置条件 | 前置条件明确且可操作 |
| 步骤可执行 | 步骤描述无歧义，新人可复现 |
| 预期可验证 | 预期结果有明确的数据/状态可断言 |
| 后置清理 | 不产生脏数据，或明确标注清理方式 |
| 优先级合理 | 优先级与需求重要程度匹配 |
| 设计方法 | 标注所采用的设计方法（等价类/边界值/...） |

## 七、用例维护与更新机制

### 7.1 触发场景

| 场景 | 操作 | 责任人 |
|------|------|--------|
| 新增需求 | 新增对应用例 | 开发/测试 |
| 需求变更 | 评审影响范围，更新或废弃相关用例 | 测试负责人 |
| Bug 修复 | 新增回归用例，确保不复发 | Bug 修复人 |
| 接口变更 | 更新关联接口字段，调整预期结果 | 接口变更人 |
| 用例废弃 | 标记状态为 deprecated，保留 30 天后归档 | 测试负责人 |

### 7.2 覆盖率指标

| 指标 | 目标 | 度量方式 |
|------|------|----------|
| P0 需求覆盖率 | 100% | P0 用例数 / P0 需求数 >= 1 |
| P0 用例通过率 | 100% | P0 pass / P0 total |
| P1 用例通过率 | 不低于 95% | P1 pass / P1 total |
| 代码行覆盖率 | 不低于 80% | 覆盖率工具统计 |
| 接口覆盖率 | 不低于 90% | 已测试接口数 / 总接口数 |

### 7.3 用例生命周期

draft -> review -> active -> (deprecated -> archived)
                   -> (failed -> linked_to_bug -> fixed -> active)

| 状态 | 含义 |
|------|------|
| draft | 编写中，尚未提交评审 |
| review | 评审中，等待审批 |
| active | 已合入用例库，可执行 |
| deprecated | 已废弃，保留 30 天后归档 |
| archived | 已归档，不可执行，仅保留历史记录 |
| failed | 执行失败，关联 Bug |
