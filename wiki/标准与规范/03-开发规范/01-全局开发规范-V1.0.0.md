## 一、工程目录结构

```
cmd/xyfamily/
  main.go                    # 入口，启动 HTTP 服务 + 迁移 + 系统初始化

internal/
  handler/                   # API 入口（按模块分包：auth/, account/, org/, team/, group/, rbac/, audit/, admin/）
  service/                   # 业务逻辑（按模块分包，与 handler 一一对应）
  repository/                # 数据访问层（pg.go, redis.go, audit.go）
  model/                     # 领域模型（struct 定义）
  middleware/                # 横切中间件（auth.go, rate_limit.go, permission.go, logging.go）
  init/                      # 系统初始化（roles/permissions, SuperAdmin, 迁移）

pkg/
  jwt/                       # JWT 封装（签发、解析、黑名单校验）
  bcrypt/                    # 密码哈希封装（bcrypt cost=12）
  response/                  # 统一响应格式（Response 结构体、错误响应）
  errors/                    # 业务错误码定义（errno.go）
  logger/                    # 结构化 JSON 日志

configs/
  config.yaml                # 配置文件（环境变量覆盖）

migrations/
  V001__init.sql             # 数据库迁移（按版本排序，升序执行）
  V002__...sql               # 后续增量迁移

test/
  auth_test.go               # 认证模块测试
  integration_test.go        # 集成测试
```

**原则**：
- 按功能模块分包（handler/auth/、service/auth/），不按技术类型分包
- `internal/` 包禁止被外部引用
- `pkg/` 包仅放可复用的工具函数

## 二、分层职责与调用规则

```
Handler → Service → Repository → Model → DB/Redis
```

| 层 | 职责 | 禁止 |
|----|------|------|
| **Handler** | 接收 HTTP 请求，参数校验，调用 Service，返回响应 | 直接调用 Repository、直接操作 DB |
| **Service** | 业务逻辑编排，调用 Repository，处理业务错误 | 直接操作 HTTP 对象、直接返回 HTTP Status |
| **Repository** | 数据访问（SQL 构造、Redis 操作），封装 ORM/GORM | 包含业务逻辑、直接操作 HTTP 对象 |
| **Model** | 领域模型 struct、DB 结构体定义 | 包含业务逻辑、直接访问外部系统 |

**禁止跨层调用**：Handler 不能直接调用 Repository，Service 不能直接调用 Handler。

## 三、错误处理规范

### 3.1 业务错误码体系

见 [接口规范](../02-需求与产品设计/02.02-产品PRD/13-接口需求规范/接口需求规范-V1.0.0.md) 2.2 节。

业务错误码定义在 `pkg/errors/errno.go`：

```go
// pkg/errors/errno.go

// 业务错误码
const (
  // 认证模块 10xxx
  ErrVcodeInvalid = 10001
  ErrVcodeFrequent = 10002
  ErrAccountNotFound = 10003
  ErrPasswordWrong = 10004

  // 账号模块 20xxx
  ErrAccountDeactivated = 20001
  ErrAccountIncomplete = 20002

  // 组织模块 30xxx
  ErrOrgNotFound = 30001
  ErrOrgNotBelong = 30002
  ErrOrgMemberFull = 30003

  // 权限模块 60xxx
  ErrPermissionDenied = 60001
  ErrLastAdminCannotDowngrade = 60002

  // 系统 80xxx
  ErrServiceUnavailable = 80001
  ErrRateLimited = 80002
  ErrInternal = 80003
)

// Error 结构体
type Error struct {
  Code    int    `json:"code"`
  Message string `json:"message"`
  Detail  string `json:"detail,omitempty"`
}

func (e *Error) Error() string { return e.Message }
```

### 3.2 错误传播规则

```
Handler 层：接收业务 Error，映射到 HTTP Status，返回统一响应格式
Service 层：返回 *errors.Error 或直接返回业务错误
Repository 层：DB/Redis 错误包装为业务 Error（如 ErrInternal）
```

**原则**：
- 错误在每一层都明确传播，不吞掉
- UI 层面使用 `message` 字段（用户友好），`detail` 字段供调试（不含敏感信息）
- 生产环境不向客户端暴露 `detail` 中的内部信息

## 四、日志规范

### 4.1 日志格式

**结构化 JSON 日志**，每行一个 JSON 对象：

```json
{
  "level": "info",
  "time": "2026-07-12T00:00:00Z",
  "logger": "handler/auth.login",
  "message": "login success",
  "account_id": "acc_xxx",
  "ip": "1.2.3.4"
}
```

### 4.2 日志字段

| 字段 | 说明 | 必填 |
|------|------|------|
| `level` | 日志级别（debug/info/warn/error） | 是 |
| `time` | 时间戳（ISO 8601, UTC） | 是 |
| `logger` | 日志来源（包名/文件名） | 是 |
| `message` | 日志消息 | 是 |
| `account_id` | 操作者账号 ID（有上下文时） | 否 |
| `org_id` | 组织上下文 ID（有上下文时） | 否 |
| `ip` | 客户端 IP（有上下文时） | 否 |
| `error` | 错误信息（错误级别时） | 否 |

### 4.3 脱敏规则

**禁止在日志中打印以下字段明文**：
- 密码（任何形式）
- 完整手机号（脱敏：`138****1234`）
- 完整邮箱（脱敏：`z***@example.com`）
- Token（任何形式，包括 jti）

## 五、配置管理

### 5.1 配置层级

```
环境变量（最高优先级）
    ↓ 覆盖
配置文件 configs/config.yaml
    ↓ 覆盖
代码默认值（最低优先级）
```

### 5.2 核心配置项

| 配置项 | 环境变量 | 默认值 | 说明 |
|--------|---------|--------|------|
| 数据库地址 | `XY_DB_HOST` | localhost | PostgreSQL 连接地址 |
| 数据库端口 | `XY_DB_PORT` | 5432 | |
| 数据库名 | `XY_DB_NAME` | xyfamily | |
| 数据库用户 | `XY_DB_USER` | - | 必填 |
| 数据库密码 | `XY_DB_PASSWORD` | - | 必填 |
| Redis 地址 | `XY_REDIS_ADDR` | localhost:6379 | |
| Redis 密码 | `XY_REDIS_PASSWORD` | - | 可选 |
| JWT 密钥 | `XY_jwt_SECRET` | - | 必填，生产环境必须设置 |
| 日志级别 | `XY_LOG_LEVEL` | info | debug/info/warn/error |
| 服务端口 | `XY_HTTP_PORT` | 8080 | |
| 登录限流阈值 | `XY_RATE_LIMIT` | 5 | 同一 IP 5 分钟失败次数 |
| 验证码有效期 | `XY_VCODE_TTL` | 300 | 秒 |
| Access Token 有效期 | `XY_ACCESS_TTL` | 1800 | 秒（30 分钟） |
| Refresh Token 有效期 | `XY_REFRESH_TTL` | 604800 | 秒（7 天） |

### 5.3 敏感配置

- JWT 密钥、数据库密码、Redis 密码 **不得写入配置文件**，必须通过环境变量注入
- 生产环境密钥通过密钥管理系统（Vault/KMS）注入
