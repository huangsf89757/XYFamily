## 一、核心设计原则

为保证 Wiki 长期可维护、新人快速上手、研发资料零丢失、合规可审计，确立五大核心原则：

| 原则 | 说明 |
|------|------|
| **分层闭环** | 按「立项→项目总览→需求设计→开发落地→测试交付→运维应急→迭代复盘→资产沉淀」全流程分层，完整覆盖软件研发全生命周期，无资料盲区 |
| **分类+标签双检索** | 目录做结构化层级分类，标签做多维度碎片化关联检索；单文档唯一归属目录，跨模块关联仅通过标签实现 |
| **唯一归属** | 单份资料仅归属一个一级/二级目录，禁止重复归档；跨业务、跨阶段文档仅通过标签关联，避免冗余 |
| **版本可追溯** | 所有方案、接口、数据库、变更文档留存完整版本信息，满足内审、等保、财务审计合规要求 |
| **密级分级管控** | 文档按公开/内部/机密分级，对应差异化读写权限，规范敏感数据、客户资料、密钥方案上传与对外输出流程 |

## 二、十级目录结构与维护规则

### 2.1 目录总览

| 编号 | 目录 | 定位 |
|------|------|------|
| 00 | 项目总览 | 公共基础区，项目入口、全局说明、团队信息 |
| 01 | 立项与项目管理前置 | 立项材料、预算、资源管理、外包交付管控 |
| 02 | 需求与产品设计 | 业务需求、PRD、原型设计、需求复盘、运营客诉 |
| 03 | 技术架构与方案设计 | 架构设计、数据库、中间件、ADR、容灾多活 |
| 04 | 开发规范与编码手册 | 命名规范、代码规范、Git 规范、工具规范 |
| 05 | 接口与模块落地文档 | 接口汇总、模块文档、数据流转、版本兼容 |
| 06 | 测试与质量保障 | 测试计划、用例库、Bug 管理、自动化、灰度 |
| 07 | 部署运维与应急故障 | 环境配置、部署发布、运维手册、故障应急 |
| 08 | 合规安全与数据治理 | 数据合规、分级治理、安全基线、内控审计 |
| 09 | 迭代复盘与业务沉淀 | 版本记录、周期复盘、踩坑沉淀、优化规划 |
| 10 | 团队资产与培训模板 | 文档模板、公共资产、新人培训、技术分享 |

### 2.2 目录维护规则

| 规则 | 说明 |
|------|------|
| **目录结构固定** | 一级大类（00~10）和二级分类**不允许擅自新增/删除/重命名**，需项目负责人 + 技术负责人双审批 |
| **三级细分可扩展** | 三级目录下可根据业务需要新增子文件夹，但需在对应一级目录的「Wiki使用全规范」中补充说明 |
| **空目录保留** | 所有目录（含暂无文档的）须保留 `.gitkeep` 文件，确保目录结构完整可见 |
| **文档唯一归属** | 每份文档仅存放在一个目录下，跨目录关联仅通过标签和正文链接实现 |

### 2.3 新增文档归档规则

新增文档时，按以下步骤确定归档位置：

```
1. 确定研发阶段 → 找到一级目录编号
2. 确定文档类型 → 找到二级分类
3. 放入对应二级目录
4. 绑定 1~3 个标签
5. 填写文档头部元数据
```

**示例：** 新写一份数据库表结构设计文档
→ 研发阶段 = 架构设计 → 一级目录 = `03-技术架构与方案设计`
→ 文档类型 = 数据库设计 → 二级目录 = `03.02-数据库设计`
→ 归档路径 = `03-技术架构与方案设计/03.02-数据库设计/数据库表结构设计-V1.0.0.md`

## 三、标签使用规范

### 3.1 六大标签维度

| 维度 | 用途 | 标签示例 |
|------|------|----------|
| **业务维度** | 区分业务模块 | 认证模块、账号模块、组织模块、团队模块、小组模块、权限模块、审计模块、系统基础 |
| **研发阶段** | 区分生命周期 | 立项阶段、需求阶段、设计阶段、开发阶段、测试阶段、灰度发布、交付阶段、运维阶段、迭代优化、复盘沉淀 |
| **文档类型** | 区分资料属性 | 立项文档、需求PRD、技术方案、ADR决策、接口文档、数据库设计、部署手册、测试用例、故障复盘、规范标准、合规报告、培训课件 |
| **状态/优先级** | 管控有效性 | 核心文档、普通文档、临时草稿、待更新、正式生效、已废弃、机密文档、对外交付版 |
| **职能角色** | 区分使用人群 | 产品、后端、安卓开发、iOS开发、H5前端、小程序开发、Web前端、测试、运维、项目管理、运营、客服、安全 |
| **合规&技术专项** | 安全/分布式场景 | 数据脱敏、等保合规、隐私治理、漏洞安全、分布式、微服务、中间件、容灾多活、第三方集成 |

### 3.2 标签使用规则

| 规则 | 说明 |
|------|------|
| **强制绑定** | 每篇文档**必须绑定 1~3 个标签**，不绑定标签的文档视为不合规 |
| **统一标签池** | 仅允许使用上方列出的标签，**禁止自定义零散标签** |
| **优先绑定文档类型** | 3 个标签中至少 1 个来自「文档类型」维度 |
| **标签选择建议** | 优先绑定「文档类型」+「业务维度」或「文档类型」+「研发阶段」|

### 3.3 标签示例

| 文档 | 推荐标签 |
|------|----------|
| PRD 文档 | 需求PRD、权限模块 |
| 数据库设计文档 | 数据库设计、设计阶段 |
| 部署手册 | 部署手册、交付阶段 |
| 故障复盘 | 故障复盘、复盘沉淀 |
| ADR 决策记录 | ADR决策、设计阶段 |
| 安全合规报告 | 合规报告、等保合规 |

## 四、文档命名规范

### 4.1 命名格式

```
文档名称-V版本.md
```

- 密级、日期、模块、文档类型等信息由**目录结构**和**文档头部元数据**承载，文件名不重复

### 4.2 命名要素

| 要素 | 说明 |
|------|------|
| 文档名称 | 简洁明了，能区分同目录下不同文档 |
| 版本号 | 语义化版本（如 V1.0.0），同目录存在多版本时便于区分 |

### 4.3 命名示例

```
02-需求与产品设计/02.02-产品PRD/PRD-V1.0.0.md
03-技术架构与方案设计/03.05-ADR架构决策记录/数据库选型决策-V1.0.0.md
07-部署运维与应急故障/07.02-部署发布手册/首次部署手册-V1.0.0.md
07-部署运维与应急故障/07.06-故障复盘台账/支付服务宕机复盘-V1.0.0.md
```

### 4.4 命名禁止项

- 禁止使用日期、密级、模块名作为文件名前缀（这些信息在元数据中体现）
- 禁止使用空格、特殊字符（仅允许中文、英文、数字、连字符）
- 禁止使用「副本」「final」「最终版」等无意义后缀

## 六、变更人填写规范

文档的「编写人」和「变更记录-变更人」字段，统一使用**当前实际操作所使用的 AI 编程工具名称**，便于追溯每条变更的来源：

| 使用工具 | 变更人填写 |
|----------|-----------|
| Claude Code | `ClaudeCode` |
| CatPaw | `CatPaw` |
| Trae | `Trae` |
| WorkBuddy | `WorkBuddy` |
| 人工编辑 | 编辑者真实姓名或工号 |

**规则：**
- 谁创建/修改了文档，就填谁的名称
- 同一文档的不同版本记录可出现不同变更人（如 V1.0.0 由 CatPaw 创建，V1.1.0 由 Trae 修改）
- 纯人工编辑时填写真实姓名或工号

## 七、文档版本管理规则

### 7.1 语义化版本规则

文档版本号采用 `V主版本.次版本.修订号` 格式：

| 版本变更 | 触发条件 | 示例 |
|----------|----------|------|
| **主版本** (V2.0.0) | 文档内容重大重构、方向性变更 | PRD 从单体架构改为微服务架构 |
| **次版本** (V1.1.0) | 新增章节、新增功能需求、较大内容更新 | PRD 新增审计日志模块 |
| **修订号** (V1.0.1) | 修正错误、补充细节、调整措辞 | 修正接口路径拼写、补充验收标准 |

### 7.2 版本更新规则

| 规则 | 说明 |
|------|------|
| 每次修改必须递增版本号 | 不允许修改内容但不升版本 |
| 变更记录必须同步追加 | 每次升版本必须在变更记录表新增一行 |
| 旧版本处理 | 原则上只保留最新版本；重大版本更迭时可在同目录保留旧版本（如 `PRD-V1.0.0.md` + `PRD-V2.0.0.md`） |
| 版本号不可回退 | 已发布的版本号不可复用或回退 |

### 7.3 版本状态流转

```
临时草稿 → 正式生效 → 待更新 → 正式生效（循环）
                 ↘ 已废弃
```

- **临时草稿**：新建文档初始状态
- **正式生效**：经审核人审核通过后生效
- **待更新**：内容过期但尚未更新
- **已废弃**：不再使用，保留归档不删除

## 八、文档编写模板

### 8.1 标准文档模板

所有正式文档需按以下模板编写：

```markdown
# 文档标题

> 一句话描述文档用途

---

## 文档信息

| 项目 | 内容 |
|------|------|
| 文档密级 | 内部 |
| 文档版本 | V1.0.0 |
| 编写人 | CatPaw |
| 审核人 | - |
| 生效时间 | 2026-07-10 |
| 废弃时间 | - |
| 关联标签 | 标签1、标签2 |
| 关联目录 | XX-目录名/XX.XX-子目录名 |

## 变更记录

| 版本 | 日期 | 变更内容 | 变更人 |
|------|------|----------|--------|
| V1.0.0 | 2026-07-10 | 初始创建 | CatPaw |

---

## 正文内容

（文档实际内容）
```

### 8.2 模板使用说明

| 要素 | 规则 |
|------|------|
| 文档标题 | `#` 一级标题，简洁明确 |
| 引用描述 | `>` 引用块，一句话说明文档用途 |
| 文档信息表 | 必须紧跟 `---` 分割线后，格式固定 |
| 变更记录表 | 紧跟文档信息表后，每次变更追加一行 |
| 正文分割 | 变更记录与正文之间用 `---` 分割 |
| 章节编号 | 正文章节建议编号（一、二、三...）便于引用 |

## 九、文档更新机制

| 机制 | 规则 | 责任人 |
|------|------|--------|
| **迭代更新** | 版本上线 3 个工作日内同步更新对应接口、方案、部署文档 | 对应模块开发 |
| **变更同步** | 需求、架构、接口、线上配置变更，**当日同步更新** Wiki | 变更发起人 |
| **季度盘点** | 每季度末统一清理已废弃文档、更新过期资料、归档历史版本 | 技术负责人 |
| **机密文档管控** | 密钥、数据库账号、客户隐私方案禁止上传明文，需脱敏处理 | 安全负责人 |

### 9.1 文档更新检查清单

每次更新文档时，检查以下项：

- [ ] 版本号是否递增
- [ ] 变更记录是否追加
- [ ] 变更人是否正确填写
- [ ] 关联标签是否仍然准确
- [ ] 正文中的交叉引用链接是否有效
- [ ] 机密信息是否已脱敏

## 十、密级分级管控

### 10.1 密级定义

| 密级 | 标识 | 说明 | 可见范围 |
|------|------|------|----------|
| **公开** | 公开 | 可对外公开的文档 | 所有人（含外部） |
| **内部** | 内部 | 仅项目团队内部可见 | 全体项目成员 |
| **机密** | 机密 | 涉及密钥、客户隐私、核心架构等敏感信息 | 仅项目负责人 + 安全负责人 |

### 10.2 密级管理规则

| 规则 | 说明 |
|------|------|
| 默认密级 | 新建文档默认密级为「内部」 |
| 密级变更 | 需项目负责人或安全负责人审批 |
| 机密文档存储 | 禁止在 Wiki 中上传明文密钥、密码、证书；需脱敏后存储，原始值存放于密钥管理系统 |
| 机密文档检索 | 不出现在公开搜索结果中，不参与对外导出 |

### 10.3 敏感信息脱敏规则

| 信息类型 | 脱敏方式 | 示例 |
|----------|----------|------|
| 数据库连接串 | 替换为占位符 | `postgresql://***:***@***:5432/xyfamily` |
| API 密钥 | 替换为占位符 | `sk-****************************` |
| 用户手机号 | 中间四位脱敏 | `138****1234` |
| 用户邮箱 | 用户名部分脱敏 | `z***@example.com` |
| IP 地址 | 末位脱敏 | `192.168.1.***` |

## 十一、权限管控规范

### 11.1 分级权限

| 权限级别 | 范围 | 说明 |
|----------|------|------|
| 只读权限 | 全员 | 可查看公开、内部文档，检索学习；机密文档无查看权限 |
| 编辑权限 | 模块负责人 | 对应模块产品、开发、测试、运维，仅可编辑自身业务文档 |
| 管理权限 | 项目/技术/安全负责人 | 目录调整、文档审核、废弃清理、密级变更 |
| 对外输出权限 | 项目负责人 + 安全负责人 | 双审批后，可导出对外交付文档 |

### 11.2 权限管理规则

| 规则 | 说明 |
|------|------|
| 最小权限原则 | 仅授予完成工作所需的最低权限 |
| 权限变更记录 | 所有权限变更需记录在 `08.05-内控审计资料` |
| 离场回收 | 人员离职/转岗后 1 个工作日内回收 Wiki 权限 |
| 外包人员管控 | 外包人员仅授予只读权限，特殊情况需项目负责人审批 |

## 十二、对外交付文档规范

### 12.1 交付文档管理

| 规则 | 说明 |
|------|------|
| 存放位置 | 统一存放 `01.04-交付管控规范` |
| 脱敏要求 | 必须脱敏内部密钥、数据库地址、内部架构细节 |
| 标签标记 | 必须标记「对外交付版」标签 |
| 审批流程 | 项目负责人 + 安全负责人双审批后方可对外提供 |

### 12.2 交付文档检查清单
