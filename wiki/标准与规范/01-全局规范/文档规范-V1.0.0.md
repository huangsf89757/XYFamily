## 一、文档头部信息

### 1.1 必填字段

所有正式文档**必须**在正文前包含以下信息：

| 字段 | 说明 | 示例 |
|------|------|------|
| 文档密级 | 公开 / 内部 / 机密 | 内部 |
| 文档版本 | 语义化版本号 | V1.0.0 |
| 编写人 | 文档编写者（规则见 1.3） | CatPaw |
| 审核人 | 文档审核者 | 张三 |
| 生效时间 | 文档生效日期 | 2026-07-10 |
| 废弃时间 | 文档废弃日期，未废弃填 `-` | - |
| 关联标签 | 1~3 个标签 | 需求PRD、权限模块 |
| 关联目录 | 所属目录路径 | 02-需求与产品设计/02.02-产品PRD |

### 1.2 变更记录

每篇文档必须包含变更记录表：

| 列名 | 说明 |
|------|------|
| 版本 | 本次变更对应的版本号 |
| 日期 | 变更日期（YYYY-MM-DD） |
| 变更内容 | 简述本次变更内容 |
| 变更人 | 变更操作人（规则见 1.3） |

### 1.3 编写人 / 变更人填写规范

文档的「编写人」和「变更记录-变更人」字段，统一使用**当前实际操作所使用的 AI 编程工具名称**，便于追溯每条变更的来源：

| 使用工具 | 填写值 |
|----------|--------|
| Claude Code | `ClaudeCode` |
| CatPaw | `CatPaw` |
| Trae | `Trae` |
| WorkBuddy | `WorkBuddy` |
| 人工编辑 | 编辑者真实姓名或工号 |

**规则：**
- 谁创建/修改了文档，就填谁的名称
- 同一文档的不同版本记录可出现不同变更人（如 V1.0.0 由 CatPaw 创建，V1.1.0 由 Trae 修改）
- 纯人工编辑时填写真实姓名或工号

## 二、文档命名规范

### 2.1 命名格式

```
编号-名称/
```

- 文件夹命名：`编号-名称`（如 `01-产品PRD`）
- 文档命名：`编号-文档名称-V版本.md`（如 `01-注册认证-V1.0.0.md`）
- 编号仅在当前父级目录内递增，不继承上级编号
- 密级、日期、模块、文档类型等信息由**目录结构**和**文档头部信息**承载，文件名不重复

### 2.2 命名要素

| 要素 | 说明 |
|------|------|
| 编号 | 当前目录内的顺序编号，从 `01` 开始递增 | 01、02、03... |
| 名称 | 简洁明了，能区分同级文件夹或同目录下不同文档 | 产品PRD、注册认证 |
| 版本号 | 语义化版本（如 V1.0.0），同目录存在多版本时便于区分 | V1.0.0 |

### 2.3 命名示例

**文件夹示例：**

```
02-需求与产品设计/
  ├── 01-产品PRD/
  ├── 02-原型与UI设计/
  └── 03-需求复盘/

03-技术架构与方案设计/
  ├── 01-整体架构设计/
  ├── 02-数据库设计/
  └── 03-中间件专项方案/
```

**文档示例：**

```
02-需求与产品设计/01-产品PRD/01-注册认证-V1.0.0.md
02-需求与产品设计/01-产品PRD/02-登录认证-V1.0.0.md
03-技术架构与方案设计/02-数据库设计/01-ER图设计-V1.0.0.md
07-部署运维与应急故障/02-部署发布手册/01-首次部署手册-V1.0.0.md
```

### 2.4 命名禁止项

- 禁止使用日期、密级、模块名作为文件名前缀（这些信息在头部信息中体现）
- 禁止在子文件夹或文档名中重复携带父级编号（如 `02.02-产品PRD` 或 `03.05-数据库选型`）
- 禁止使用空格、特殊字符（仅允许中文、英文、数字、连字符）
- 禁止使用「副本」「final」「最终版」等无意义后缀

## 三、文档版本管理

### 3.1 语义化版本规则

文档版本号采用 `V主版本.次版本.修订号` 格式：

| 版本变更 | 触发条件 | 示例 |
|----------|----------|------|
| **主版本** (V2.0.0) | 文档内容重大重构、方向性变更 | PRD 从单体架构改为微服务架构 |
| **次版本** (V1.1.0) | 新增章节、新增功能需求、较大内容更新 | PRD 新增审计日志模块 |
| **修订号** (V1.0.1) | 修正错误、补充细节、调整措辞 | 修正接口路径拼写、补充验收标准 |

### 3.2 版本更新规则

| 规则 | 说明 |
|------|------|
| 每次修改必须递增版本号 | 不允许修改内容但不升版本 |
| 变更记录必须同步追加 | 每次升版本必须在变更记录表新增一行 |
| 旧版本处理 | 原则上只保留最新版本；重大版本更迭时可在同目录保留旧版本（如 `PRD-V1.0.0.md` + `PRD-V2.0.0.md`） |
| 版本号不可回退 | 已发布的版本号不可复用或回退 |

## 四、文档编写模板

### 4.1 标准模板

所有正式文档需按以下模板编写：

```markdown
# 文档标题

> 一句话描述文档用途

---

## 文档信息

| 项目 | 内容 |
|------|------|
| 文档密级 | 内部 |
| 文档版本 | V1.0.0 |
| 编写人 | CatPaw |
| 审核人 | - |
| 生效时间 | 2026-07-10 |
| 废弃时间 | - |
| 关联标签 | 标签1、标签2 |
| 关联目录 | XX-目录名/XX.XX-子目录名 |

## 变更记录

| 版本 | 日期 | 变更内容 | 变更人 |
|------|------|----------|--------|
| V1.0.0 | 2026-07-10 | 初始创建 | CatPaw |

---

## 正文内容

（文档实际内容）
```

### 4.2 模板使用说明

| 要素 | 规则 |
|------|------|
| 文档标题 | `#` 一级标题，简洁明确 |
| 引用描述 | `>` 引用块，一句话说明文档用途 |
| 文档信息表 | 必须紧跟 `---` 分割线后，格式固定 |
| 变更记录表 | 紧跟文档信息表后，每次变更追加一行 |
| 正文分割 | 变更记录与正文之间用 `---` 分割 |
| 章节编号 | 正文章节建议编号（一、二、三...）便于引用 |

---

## 五、文档更新检查清单

每次更新文档时，检查以下项：

- [ ] 版本号是否递增
- [ ] 变更记录是否追加
- [ ] 变更人是否正确填写
- [ ] 关联标签是否仍然准确
- [ ] 正文中的交叉引用链接是否有效
- [ ] 机密信息是否已脱敏
