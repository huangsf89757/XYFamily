## 一、命名规范

### 1.1 包名

- 小写单词，不使用下划线和驼峰
- 短小且语义明确
- 示例：`handler/auth`、`service/org`、`repository/pg`、`pkg/errors`

### 1.2 文件名

- 小写，使用下划线分隔
- 示例：`login_handler.go`、`permission_checker.go`、`role_initializer.go`

### 1.3 函数名

- 大驼峰（PascalCase），与 Go 惯用法一致
- 动词开头，语义明确
- 示例：`CreateOrganization`、`CheckPermission`、`ValidateMembership`

### 1.4 变量名

- 小驼峰（camelCase）
- 局部变量简短明确，如 `account`、`ctx`、`err`
- 布尔变量使用 `is`/`has`/`should`/`can` 前缀，如 `isAdmin`、`hasPermission`、`isDeactivated`

### 1.5 常量名

- 大驼峰（PascalCase）
- 示例：`MaxPageSize`、`DefaultPageSize`、`VcodeTTL`

### 1.6 接口名

- 动词 + `er` 形式，如 `Repository`、`Handler`、`Logger`
- 或语义明确的名词，如 `AccountService`

## 二、Go 惯用法

### 2.1 Error 返回

```go
// 正确：函数返回 error
func CreateAccount(ctx context.Context, req *CreateAccountReq) (*Account, error) {
    if req.Phone == "" && req.Email == "" {
        return nil, errors.BadRequest("至少需要手机号或邮箱")
    }
    // ...
    return account, nil
}

// 错误：panic 或忽略 error
```

### 2.2 Context 传递

```go
// 正确：context 作为第一个参数
func Login(ctx context.Context, req *LoginReq) (*AuthResult, error) {
    // 使用 ctx 传递超时、取消信号
    account, err := repo.FindAccount(ctx, req.Account)
    if err != nil {
        return nil, err
    }
    // ...
}

// 错误：不使用 context
func Login(req *LoginReq) (*AuthResult, error) {
    // ...
}
```

### 2.3 接口抽象

```go
// 正确：Repository 接口
type AccountRepository interface {
    FindByAccountID(ctx context.Context, accountID string) (*Account, error)
    FindByPhone(ctx context.Context, phone string) (*Account, error)
    FindByEmail(ctx context.Context, email string) (*Account, error)
    Create(ctx context.Context, account *Account) error
    Update(ctx context.Context, account *Account) error
}
```

### 2.4 指针 vs 值

- struct 较大或需要修改时使用指针：`*Account`、`*Organization`
- 简单的 struct（如 DTO）使用值：`LoginReq`、`Response`
- 切片和 map 本身就是引用类型，不需要指针

## 三、数据库操作规范

### 3.1 使用 GORM

```go
// 正确：使用 GORM 操作
func (r *PGAccountRepository) FindByAccountID(ctx context.Context, accountID string) (*model.Account, error) {
    var account model.Account
    err := r.db.WithContext(ctx).Where("account_id = ?", accountID).First(&account).Error
    if err != nil {
        return nil, err
    }
    return &account, nil
}

// 错误：使用 sql.DB 原生操作
```

### 3.2 事务边界

- 事务边界在 Service 层
- 一个业务操作对应一个事务，不跨 Service 层
- 事务内操作尽量简洁，避免长时间持有锁

```go
// 正确：Service 层事务
func (s *OrgService) InviteMember(ctx context.Context, orgID string, req *InviteMemberReq) error {
    return tx.WithTransaction(ctx, func(txn *gorm.DB) error {
        // 创建邀请记录
        err := s.repo.CreateInvitation(txn, &invitation)
        if err != nil {
            return err
        }
        // 记录审计日志
        return s.auditRepo.Log(txn, &auditLog)
    })
}

// 错误：Repository 层事务
```

### 3.3 连接池配置

```go
// 数据库连接池：最小 10，最大 100
db, err := gorm.Open(postgres.Open(dsn), &gorm.Config{
    // ...
})
sqlDB, _ := db.DB()
sqlDB.SetMaxOpenConns(100)
sqlDB.SetMaxIdleConns(10)
sqlDB.SetConnMaxLifetime(time.Hour)
```

## 四、单元测试规范

### 4.1 测试覆盖率

- **最小覆盖率：80%**
- Service 层和 Repository 层必须覆盖
- Handler 层覆盖关键路径

### 4.2 测试文件

- 测试文件与源代码同目录
- 命名：`xxx_test.go`（源文件）→ `xxx_test.go`（测试文件）

### 4.3 AAA 模式

```go
func TestCreateAccount(t *testing.T) {
    // Arrange
    req := &CreateAccountReq{Phone: "13800138000", Code: "123456"}
    repo := &mockAccountRepository{}

    // Act
    svc := NewAccountService(repo)
    account, err := svc.CreateAccount(context.Background(), req)

    // Assert
    assert.NoError(t, err)
    assert.Equal(t, "13800138000", account.Phone)
}
```

### 4.4 Mock 策略

- Repository 层使用 mock 对象
- 使用 `gomock` 或自定 mock 实现
- 测试文件放在 `_test` 目录或使用 build tag

```go
// mock 示例
type mockAccountRepository struct {
    findFunc func(ctx context.Context, phone string) (*model.Account, error)
}

func (m *mockAccountRepository) FindByPhone(ctx context.Context, phone string) (*model.Account, error) {
    return m.findFunc(ctx, phone)
}
```
