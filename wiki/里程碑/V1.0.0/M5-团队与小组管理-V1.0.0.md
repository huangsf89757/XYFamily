## 四、技术要点

### 4.1 层级关系

```
组织 (Organization)
  └── 团队 (Team) — 属于组织
       └── 小组 (Group) — 属于团队
```

- 团队必须属于一个组织
- 小组必须属于一个团队
- 查询时强制带上上级 ID 条件

### 4.2 软删除

- teams 表: `deleted_at` 列，归档后标记
- groups 表: `deleted_at` 列，删除后标记
- 唯一索引条件: `WHERE deleted_at IS NULL`（允许同名重新创建）

### 4.3 角色范围

| 层级 | 可分配角色 |
|------|-----------|
| 团队 | team_core_admin / team_ordinary_admin / RegularMember |
| 小组 | group_core_admin / group_ordinary_admin / RegularMember |

### 4.4 接棒机制

创建团队/小组时，可通过 `initial_admin` 字段指定初始管理员：
- 如果指定，被指定人自动成为该层级 Admin
- 创建者也可以自动成为 Admin（如果未指定 initial_admin）
- 接棒 = 创建者将管理权移交给 initial_admin

## 六、风险评估

| 风险 | 等级 | 应对措施 |
|------|------|---------|
| 层级查询性能 | 中 | org_id + team_id 复合索引 |
| 归档团队下小组处理 | 低 | 归档团队时仅标记 deleted_at，不影响小组数据 |
| 邀请流复用复杂度 | 中 | 统一 invitation_target_type 区分层级 |
