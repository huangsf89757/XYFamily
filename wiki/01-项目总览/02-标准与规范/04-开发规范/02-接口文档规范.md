## 一、接口文档编写要求

每个接口文档**必须**包含以下 5 个核心章节，缺一不可：

| # | 章节 | 必要 | 说明 |
|---|------|------|------|
| ① | 接口描述 | ✅ 必须 | 名称、功能说明、HTTP 方法、路径、认证方式、限流策略 |
| ② | 请求体 | ✅ 必须 | 参数定义（字段名、类型、必填性、约束、示例） |
| ③ | 响应体 | ✅ 必须 | 成功响应（标准信封格式）+ 失败响应（典型错误） |
| ④ | 错误码 | ✅ 必须 | 该接口可能返回的所有错误码及其 HTTP 状态 |
| ⑤ | 示例 | ✅ 必须 | 完整的请求/响应 JSON 示例（含边界场景） |

## 二、接口描述规范

### 2.1 基本信息表

每个接口必须在开头提供基本信息表，格式如下：

| 项目 | 内容 |
|------|------|
| 方法 | `GET` / `POST` / `PATCH` / `PUT` / `DELETE` |
| 路径 | 完整路径（含 Base Path `/api/v1/`） |
| 认证 | ✅ 需 Token / ❌ 公开 |
| HTTP Status | 成功时返回的 HTTP 状态码 |
| 限流 | 限流规则（如无则注明"无"） |

### 2.2 接口命名规范

| 维度 | 规范 |
|------|------|
| 资源路径 | 使用 `snake_case`，组织用缩写 `/org`，团队/小组用复数 `/teams`/`/groups` |
| 路径参数 | `snake_case`，ID 后缀 `_id`（如 `:org_id`、`:team_id`） |
| 层级体现 | 体现 组织→团队→小组 的层次关系 |

**层级路径规范**：

| 层级 | 路径格式 | 示例 |
|------|----------|------|
| 组织 | `/org`、`/org/:org_id` | `/api/v1/org/:org_id` |
| 团队 | `/org/:org_id/teams/:team_id` | `/api/v1/org/:org_id/teams/:team_id` |
| 小组 | `/org/:org_id/teams/:team_id/groups/:group_id` | `/api/v1/org/:org_id/teams/:team_id/groups/:group_id` |

### 2.3 HTTP 方法规范

| 方法 | 语义 | 使用场景 |
|------|------|----------|
| `POST` | 创建资源或执行动作 | 新建资源、认证操作、不可逆动作 |
| `GET` | 查询资源 | 读取数据、列表查询（**不得包含写操作**） |
| `PATCH` | 部分更新 | 修改部分字段（仅传递需要变更的字段） |
| `PUT` | 全量替换 | 替换整个资源值（如角色变更，全量替换角色值） |
| `DELETE` | 删除资源 | 软删除（使用 `deleted_at` 标记） |

> **规则**：角色变更接口使用 `PUT`（全量替换），其余更新接口使用 `PATCH`（部分更新）。

### 2.4 认证方式

| 方式 | 适用场景 | 说明 |
|------|----------|------|
| 公开接口 | 注册、登录、重置密码、发送验证码、健康检查 | 无需 Token，标记 `❌` |
| Bearer Token | 所有其他接口 | `Authorization: Bearer <access_token>`，标记 `✅` |

## 三、请求体规范

### 3.1 参数定义表格

每个接口必须使用以下格式定义请求参数：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `field_name` | `type` | ✅/否/⚠️ | 字段说明、约束 |

**必填标记含义**：

| 标记 | 含义 |
|------|------|
| ✅ | 必填 |
| 否 | 可选 |
| ⚠️ | 条件必填（如 phone/email 至少一个） |

### 3.2 字段类型规范

| 类型 | 说明 | 示例 |
|------|------|------|
| `string` | 字符串 | `"zhangsan"` |
| `string` (UUID) | UUID 格式 | `"123e4567-e89b-12d3-a456-426614174000"` |
| `string` (枚举) | 枚举值（明确列出所有合法值） | `"organization_core_admin"` |
| `int` | 整数 | `20` |
| `boolean` | 布尔值 | `true` |
| `array` | 数组 | `["uuid-1", "uuid-2"]` |

### 3.3 字段约束要求

- **字符串长度**：明确最小/最大长度（如 `minLength: 12`）
- **正则模式**：对格式敏感的字段（手机号 11 位、邮箱 RFC 5322）必须定义正则
- **枚举值**：明确列出所有合法值，不得省略
- **条件必填**：明确说明条件（如"phone/email 至少提供一个"）

### 3.4 请求示例要求

每个接口必须提供完整的请求 JSON 示例，包含所有必填字段和常见可选字段。

## 四、响应体规范

### 4.1 标准响应信封

所有接口**必须**使用统一的 JSON 响应信封：

```json
{
  "code": 0,
  "message": "success",
  "detail": null,
  "data": {},
  "pagination": null
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | int | 业务状态码。`0`=成功；非0=失败（5 位分段体系） |
| `message` | string | 错误摘要（中文） |
| `detail` | string/null | 详细错误信息（成功时为 `null`） |
| `data` | any/null | 成功时返回业务数据；失败时为 `null` |
| `pagination` | object/null | 分页信息（非分页接口为 `null`） |

> **约束**：所有接口响应必须包含上述 5 个字段，不得遗漏或增删。

### 4.2 分页响应格式

分页接口的 `data` 字段结构：

```json
{
  "code": 0,
  "message": "success",
  "detail": null,
  "data": {
    "items": [ { "field": "value" } ],
    "total": 100,
    "page": 1,
    "page_size": 20
  },
  "pagination": null
}
```

### 4.3 错误响应格式

所有错误响应**必须**包含完整信封：

```json
{
  "code": 101002,
  "message": "Token 无效或已过期",
  "detail": "Token 签名验证失败或已超出有效期",
  "data": null,
  "pagination": null
}
```

### 4.4 响应示例要求

- 每个接口必须提供**成功响应**和**至少一个典型错误响应**示例
- PII 数据必须脱敏：手机号 `138****1234`，邮箱 `l***@example.com`
- 时间字段必须使用 **ISO 8601 UTC** 格式（`2026-07-13T10:00:00Z`）

## 五、错误码规范

### 5.1 5 位分段体系

业务错误码采用 **5 位数字分段体系**，前 2~3 位表示模块，后 2~3 位表示具体错误：

| 模块段 | Code 范围 | 说明 |
|--------|----------|------|
| 101xxx | 认证 | Token 无效/过期、账号不存在、密码错误 |
| 104xxx | 认证限流 | 登录失败过多已锁定 |
| 110xxx | 验证码 | 验证码错误/过期/已使用/尝试超限 |
| 114xxx | 验证码限流 | 发送频率超限 |
| 20xxx | 账号 | 密码强度、账号已注册/已注销、宽限期 |
| 21xxx | 邀请 | 邀请不存在/过期、状态不允许、account_id 不匹配 |
| 30xxx | 组织 | 组织不存在、名称重复、最后管理员约束 |
| 40xxx | 团队 | 团队不存在、名称重复、最后管理员约束 |
| 50xxx | 小组 | 小组不存在、名称重复、角色不合法 |
| 60xxx | 权限 | 不在当前 scope、权限不足、Header 参数错误 |
| 80xxx | 系统 | 内部错误、Redis/数据库连接失败、服务不可用 |

> 完整错误码列表见上方各模块定义。

### 5.2 接口级错误码标注

每个接口必须在"错误码"章节列出**该接口可能返回的所有错误码**：

| HTTP | 业务 Code | 说明 |
|------|----------|------|
| 400 | 110001 | 验证码错误 |
| 401 | 101002 | Token 无效或已过期 |

### 5.3 错误码完整性要求

- 错误码必须覆盖所有可能的失败场景
- 错误码编号**全局唯一**，不得重复
- 接口文档中的错误码必须与错误码总表保持一致

## 六、示例规范

### 6.1 请求示例

每个接口必须提供完整请求 JSON 示例，包含所有必填字段和常见可选字段。

### 6.2 响应示例

每个接口必须提供：
- **成功响应**：完整信封，含实际业务数据
- **至少一个典型错误响应**：完整信封，含 `code`、`message`、`detail`

## 七、通用参数规范

### 7.1 分页参数

| 参数 | 类型 | 默认值 | 约束 | 说明 |
|------|------|--------|------|------|
| `page` | int | 1 | ≥ 1 | 页码（从 1 开始） |
| `page_size` | int | 20 | 1~100 | 每页数量（最大 100） |

### 7.2 筛选/排序参数

| 参数 | 类型 | 说明 |
|------|------|------|
| `keyword` | string | 关键词搜索 |
| `role` | string | 角色筛选 |
| `from` | string | 起始时间（ISO 8601） |
| `to` | string | 结束时间（ISO 8601） |
| `sort` | string | 排序字段（如 `created_at`） |
| `order` | string | 排序方向（`asc`/`desc`，默认 `desc`） |

### 7.3 认证 Header

| Header | 必填 | 说明 |
|--------|------|------|
| `Authorization` | 需认证接口必填 | `Bearer <access_token>` |
| `X-Organization-ID` | 组织相关接口必填 | 当前操作所属组织 ID |
| `X-Team-ID` | 团队相关接口必填 | 当前操作所属团队 ID |
| `X-Group-ID` | 小组相关接口必填 | 当前操作所属小组 ID |

## 八、安全规范

### 8.1 必须遵守的安全要求

| 要求 | 说明 |
|------|------|
| 参数校验 | 所有接口必须校验输入参数合法性，不得信任任何用户输入 |
| 敏感字段 | 密码类字段不记录到日志，响应中不包含密码 |
| PII 脱敏 | 响应中手机号 `138****1234`，邮箱 `l***@example.com` |
| HTTPS | 生产环境强制 HTTPS |
| 限流 | 验证码/登录等敏感接口必须标注限流策略 |
| Token 黑名单 | 登出/改密码后，Access Token jti 写入 Redis 黑名单 |

### 8.2 邀请接口安全

接受/拒绝邀请属于高风险操作，**必须**满足：
- 携带 `Authorization: Bearer <access_token>` Header
- Token 中 `account_id` 必须与请求体 `account_id` 及邀请目标**三者一致**
- 否则返回 403

### 8.3 Token 安全

| 场景 | 处理方式 |
|------|----------|
| 登出 | Access Token jti 写入 Redis 黑名单（TTL=Token 剩余有效期） |
| 修改密码 | 当前 Access Token 写入 Redis 黑名单，强制重新登录 |
| 密码重置 | 撤销该账号所有现有 Session |

## 九、接口文档结构模板

每个模块的接口文档**必须**按以下结构组织：

```
## X、接口名称

- 基本信息表（方法/路径/认证/HTTP Status/限流）
- 请求体定义（参数表格 + 请求示例 JSON）
- 成功响应示例（标准信封格式）
- 错误响应示例（标准信封格式）
- 错误码表（HTTP / 业务 Code / 说明）
```
