## 四、技术要点

### 4.1 JWT 策略

- Access Token: HS256, 30min, 无状态, Claims 包含 `account_id`, `org_ids`, `roles`, `jti`
- Refresh Token: UUID, 7 天, 有状态（sessions 表），Refresh Rotation（旧 token 立即撤销）
- 黑名单: `token:blacklist:{jti}` → Redis, TTL = Token 剩余有效期

### 4.2 验证码流程

```
发送 → DB 写入 pending 状态 → 用户输入 → 校验(状态+有效期) → 标记 consumed → 执行业务
```

- 6 位数字，5 分钟 TTL
- 限流：单手机号 1 次/60s，同 IP 5 次/5min

### 4.3 登录审计

每次登录（无论成功/失败）同步写入 `login_audit_logs`，记录：
- `login_method`: phone_password / email_password / username_password / phone_code / email_code / user_id_password
- `login_result`: success / failure
- `failure_reason`: 密码错误 / 账号不存在 / 账号已注销 / 限流锁定
- `ip_address`, `user_agent`

## 六、风险评估

| 风险 | 等级 | 应对措施 |
|------|------|---------|
| Refresh Token 并发刷新 | 中 | DB 行锁 + 旧 token 状态校验 |
| 验证码被暴力破解 | 低 | 6 位 + 5min TTL + 限流 |
| Token 黑名单 Redis 宕机 | 中 | 黑名单 miss 时 fallback DB 查询 sessions |
