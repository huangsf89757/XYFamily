## 一、与接口设计的区别

| 维度 | 接口设计 | 标准接口 |
|------|----------|----------|
| 定位 | 设计思路文档 | 标准化接口契约 |
| 内容侧重 | 业务流程、架构决策、缓存策略、SQL、安全设计 | 接口名称、请求/响应字段定义、完整示例（含多种失败场景） |
| 读者 | 后端开发者、架构师 | 前后端开发者、测试工程师、API 集成方 |
| 示例数量 | 每个接口 1 个成功 + 1 个失败 | 每个接口 1 个成功 + 多个失败（覆盖所有错误码） |

## 二、标准接口文档模板

每个接口必须按以下结构组织：

```markdown
## N. 接口名称
> HTTP_METHOD /api/v1/path

### 简介

（一句话或一段话描述接口用途）

### 请求

#### 请求头

（需认证的接口列出 Header 表格；公开接口写"无"）

#### 请求体

（参数定义表格 + 请求示例 JSON）
（GET 请求使用查询参数；无请求体则写"无"）

### 响应

#### 响应体

（返回字段定义表格；无业务数据则写"无业务数据（data 为 null）"）

### 示例

#### 成功示例

##### 成功示例1：xxx

（完整 JSON）

##### 成功示例2：xxx

（完整 JSON，如有多个成功场景）

#### 失败示例

##### 失败示例1：xxx（HTTP 4xx）

（完整 JSON）

##### 失败示例2：xxx（HTTP 4xx）

（完整 JSON，按错误码逐个列出）

### 错误码

（该接口可能返回的所有错误码列表）

| HTTP | 业务 Code | 说明 |
|------|----------|------|
| ... | ... | ... |
```

## 四、系统接口

### 健康检查
> GET /api/v1/healthz

#### 简介

检查服务健康状态，供负载均衡和探针使用。

#### 请求

##### 请求头

无（公开接口）。

##### 请求体

无。

#### 响应

##### 响应体

| 字段 | 类型 | 说明 |
|------|------|------|
| data.status | string | `"ok"` 或 `"error"` |
| data.version | string | 服务版本号 |
| data.uptime | string | 服务运行时长 |
| data.dependencies.database | string | 数据库状态（`"ok"` / `"error"`） |
| data.dependencies.redis | string | Redis 状态（`"ok"` / `"error"`） |

#### 示例

##### 成功示例

##### 错误码

```json
{
  "code": 0,
  "message": "success",
  "detail": null,
  "data": {
    "status": "ok",
    "version": "1.0.0",
    "uptime": "72h15m30s",
    "dependencies": {
      "database": "ok",
      "redis": "ok"
    }
  },
  "pagination": null
}
```

> 依赖组件异常时，HTTP Status 仍为 200，对应字段值为 `"error"`。

##### 错误码

本接口为公开健康检查接口，除内部异常外不返回业务错误码。

| HTTP | 业务 Code | 说明 |
|------|----------|------|
| 500 | 805000 | 内部服务错误（服务不可用） |
