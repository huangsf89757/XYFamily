## 一、审查概述

### 1.1 审查方法

- **完整性审查**：逐模块核对功能需求、边界条件、异常处理、业务流程是否闭环
- **一致性审查**：跨模块术语、角色命名、权限点、状态机、数值常量是否统一
- **可行性审查**：需求在技术实现上的复杂度、风险、性能影响
- **安全审查**：认证鉴权、数据隔离、防攻击、审计合规等安全相关规则

### 1.2 审查结论（总体评估）

| 维度 | 评分 | 说明 |
|------|:--:|------|
| 需求完整性 | ⭐⭐⭐⭐☆ | 9 大模块覆盖全面，但部分边界场景缺失 |
| 文档一致性 | ⭐⭐⭐☆☆ | 跨文档存在多处术语/数值/规则不一致，需统一 |
| 安全设计 | ⭐⭐⭐⭐☆ | 安全机制较完善，但部分场景防护不足 |
| 可落地性 | ⭐⭐⭐⭐☆ | P0 需求基本可落地，部分复杂规则需技术方案细化 |
| 风险可控性 | ⭐⭐⭐☆☆ | 存在多处 P0 级风险项，需在开发前关闭 |

**总体建议**：PRD 在功能覆盖上较为完整，但存在 **17 项必须修复的不一致/缺陷** 和 **8 项建议改进项**，需在技术方案设计前完成修订。

### 🔴 DEF-011：Token 刷新并发处理描述矛盾

| 项目 | 内容 |
|------|------|
| **位置** | `01-用户认证模块/04-Token管理-V1.0.0.md` §1.1 / §4.1 |
| **问题** | 业务规则中说「仅第一个请求成功，后续请求使用新 Token 或返回未授权」，但异常处理表格中「并发刷新」的处理方式是「仅第一个请求成功，后续返回新 Token 或未授权」。这里「返回新 Token」意味着服务端需要保存新生成的 Token 并返回给并发请求，但 Token Rotation 机制要求旧 Refresh Token 立即作废，若并发请求都拿到新 Token 则会产生多个有效会话 |
| **影响** | 并发刷新机制实现复杂，且存在安全风险（多个 Refresh Token 同时有效） |
| **建议** | 明确并发刷新策略：推荐采用分布式锁保证仅一个请求成功刷新，其余请求返回「未授权，请使用新 Token 重试」或等待后重试。不应允许多个请求都获得有效的新 Refresh Token |

## 三、重要建议（P1 - 强烈建议）

### 🟡 ADV-001：增加验证码场景隔离机制

| 项目 | 内容 |
|------|------|
| **位置** | 所有涉及验证码的模块 |
| **问题** | 注册、登录、密码重置、账号恢复均使用 6 位数字验证码，但文档未明确说明各场景验证码是否隔离。若用户同时触发注册和登录验证码，两个验证码可能混淆 |
| **建议** | 明确验证码需按场景隔离（如 `register:{phone}`、`login:{phone}`、`reset:{phone}` 等不同 Redis key），避免跨场景误用 |

### 🟡 ADV-002：明确「团队普通管理员」的「小组管理」权限边界

| 项目 | 内容 |
|------|------|
| **位置** | `04-团队管理模块/03-角色与权限-V1.0.0.md` §1.1 |
| **问题** | 团队普通管理员的角色权限对比表中「管理下属小组」标记为 ✅（通过 `group.*` 继承），但团队普通管理员按定义「不可做结构性操作」。而小组的「删除」「分配角色」等属于结构性操作，团队普通管理员是否可执行？ |
| **建议** | 明确团队普通管理员对下属小组的权限：可查看小组信息和成员列表、可邀请/移除小组成员，但不可修改/删除小组、不可分配小组角色。需在权限点矩阵中细化 `group.*` 的继承范围 |

### 🟡 ADV-003：增加「组织归档」功能缺失说明

| 项目 | 内容 |
|------|------|
| **位置** | `03-组织管理模块/01-组织信息管理-V1.0.0.md` |
| **问题** | 团队有「归档」功能（FR-TEAM-009），小组有「删除」功能（FR-GROUP-003），但组织层面缺少「归档/删除」功能。若企业客户退订，SuperAdmin 如何清理组织？ |

### 🟡 ADV-004：明确账号恢复后的 Token 处理

### 🟡 ADV-005：增加「用户名修改」对登录的影响说明
