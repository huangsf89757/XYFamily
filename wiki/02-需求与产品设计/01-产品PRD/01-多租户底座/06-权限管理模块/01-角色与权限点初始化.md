## 二、业务规则

### 2.1 初始化时机与幂等

- 应在应用启动阶段执行，保证在任意业务请求到达前权限数据已就绪。
- **幂等执行**：若角色、权限点及其映射数据已存在，则跳过初始化，避免重复写入。
- 若初始化失败（如存储不可达、写入异常），应**阻断启动并告警**，不允许系统在无权限数据的情况下对外提供服务。

### 2.2 角色初始化清单

初始化 9 种角色，固定 `role_key` 与层级：

| role_key | 层级 | 角色名称 |
|----------|------|----------|
| `public` | L0 | Public（公共角色） |
| `regular_member` | L1 | RegularMember（普通成员） |
| `group_ordinary_admin` | L2 | 小组普通管理员 |
| `group_core_admin` | L3 | 小组核心管理员 |
| `team_ordinary_admin` | L4 | 团队普通管理员 |
| `team_core_admin` | L5 | 团队核心管理员 |
| `organization_ordinary_admin` | L6 | 组织普通管理员 |
| `organization_core_admin` | L7 | 组织核心管理员 |
| `super_admin` | L8 | SuperAdmin（超级管理员） |

- 组织 / 团队 / 小组的**管理员均区分核心管理员与普通管理员**：核心管理员拥有结构性 / 管理性权限，普通管理员仅负责成员管理与读取。详见 [模块 README - 角色体系](./权限管理模块.md)。
- `public` 角色对每个账号自动生效，无需额外的成员关系记录。
- `super_admin` 角色初始化后，由超级管理员初始化流程创建首个 SuperAdmin 账号并绑定该角色，详见 [超级管理员模块](../07-超级管理员模块/超级管理员模块.md)。SuperAdmin 不在任何组织 / 团队 / 小组成员关系中。

### 2.3 权限点初始化清单

初始化 45 个权限点（按域分组，采用 `domain.action` 命名）：

| 域 | 数量 | 权限点 |
|----|------|--------|
| 认证 | 5 | auth.register、auth.login、auth.reset_password、auth.logout、auth.refresh |
| 账号 | 7 | account.profile.read、account.profile.update、account.password.update、account.deactivate、account.undeactivate、account.third_party.bind、account.third_party.unbind |
| 组织 | 9 | org.create、org.read、org.update、org.member.invite、org.member.remove、org.member.list、org.role.assign、org.role.downgrade、org.team.create |
| 团队 | 9 | team.read、team.update、team.member.invite、team.member.remove、team.member.list、team.role.assign、team.role.downgrade、team.group.create、team.archive |
| 小组 | 8 | group.read、group.update、group.delete、group.member.invite、group.member.remove、group.member.list、group.role.assign、group.role.downgrade |
| 审计 | 3 | audit.login.read、audit.operation.read、audit.query |
| 超级管理员 | 4 | admin.config.read、admin.config.update、admin.force_downgrade、admin.audit.global |

> 完整的权限点矩阵与每个角色的有效权限见 [模块 README - 权限点矩阵](./权限管理模块.md)。

### 2.4 角色→权限点映射（含继承固化）

映射关系应包含两部分：

1. **基础映射**：为每个角色建立其「自身权限点」（即该角色直接持有的权限，不含继承）。
2. **继承映射**：高层角色的映射额外包含其下所有低层角色的权限点，遵循二维继承模型：
   - **纵向（范围层级）**：高层范围角色向下**只继承同层级**低层范围角色的权限。即 `organization_core_admin`（组织核心）的映射 = 自身组织域权限 + `team_core_admin`（团队核心）权限 + `group_core_admin`（小组核心）权限 + `regular_member` 权限；`organization_ordinary_admin`（组织普通）的映射 = 自身组织域权限 + `team_ordinary_admin`（团队普通）权限 + `group_ordinary_admin`（小组普通）权限 + `regular_member` 权限。
   - **横向（同范围）**：核心管理员映射额外包含同范围普通管理员权限（如 `group_core_admin` 包含 `group_ordinary_admin`）。

> 各角色的有效权限构成与示例详见 [04-权限继承](./04-权限继承.md) 与 [模块 README - 权限点矩阵](./权限管理模块.md)。

### 2.5 初始化顺序

建议执行顺序：先初始化角色，再初始化权限点，最后建立角色→权限点映射，以保证数据依赖关系成立。

## 三、边界与异常处理

| 场景 | 处理方式 |
|------|----------|
| 数据已初始化 | 幂等跳过，不重复写入 |
| 初始化写入失败 | 阻断启动，记录错误并告警 |
| 角色 / 权限点标识冲突 | 以固定标识（role_key / permission_key）为幂等键，已存在则跳过 |
| 初始化后数据不完整（部分写入） | 提供修复手段（重新执行初始化）以补全数据 |
