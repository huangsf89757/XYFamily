## 一、功能需求

### FR-PERM-004：权限继承

| 项目 | 内容 |
|------|------|
| **优先级** | P0 |
| **描述** | 高层级角色自动继承其下所有低层级角色的全部权限点 |
| **验收标准** | 高层级角色的有效权限集合 = 自身权限 + 所有低层角色权限；校验时无需运行时计算继承关系 |

## 二、业务规则

### 2.1 二维继承模型

权限继承为**二维**，由高到低：

- **范围层级（纵向）**：组织(L7/L6) > 团队(L5/L4) > 小组(L3/L2)。高层范围角色向下**只继承同层级**低层范围权限：
  - 组织核心(L7) 继承 团队核心(L5) + 小组核心(L3)
  - 组织普通(L6) 继承 团队普通(L4) + 小组普通(L2)
- **管理员层级（横向）**：同一范围内，核心管理员继承同范围普通管理员的全部权限：
  - 小组核心(L3) 继承 小组普通(L2)
  - 团队核心(L5) 继承 团队普通(L4)
  - 组织核心(L7) 继承 组织普通(L6)
- 两个维度叠加，形成由低到高的线性链：
  `Public(L0) → RegularMember(L1) → 小组普通(L2) → 小组核心(L3) → 团队普通(L4) → 团队核心(L5) → 组织普通(L6) → 组织核心(L7) → SuperAdmin(L8)`
- Public(L0) 为独立公共角色，仅持 3 个公开权限点，不参与上述继承链。
- 每个高层角色包含其下方所有角色的权限点。

> 关键约束：纵向继承**只取同层级**（如组织普通只继承团队普通 / 小组普通），避免「普通管理员越级获得低层范围核心权限」的悖论。

### 2.2 实现方式：预填充（固化继承）

- 系统在初始化角色→权限点映射时，高层角色的映射记录 = 自身权限点 + 其下所有低层角色（按上述二维线性链）的权限点。
- 运行时权限校验**只需查询当前角色的权限点集合**，无需递归计算继承关系，降低延迟与复杂度。
- 继承关系在角色初始化时固化（见 [01-角色与权限点初始化](./01-角色与权限点初始化-V1.0.0.md)）。

### 2.3 各角色有效权限构成（含继承）

| 角色 | 有效权限点构成（含继承） |
|------|--------------------------|
| Public(L0) | 3 个公开权限点 |
| RegularMember(L1) | 自身权限点 |
| 小组普通管理员(L2) | 小组普通权限 + RegularMember 权限 |
| 小组核心管理员(L3) | 小组核心权限 + 小组普通管理员权限（含 RegularMember） |
| 团队普通管理员(L4) | 团队普通权限 + 小组全部权限（普通 + 核心）+ RegularMember 权限 |
| 团队核心管理员(L5) | 团队核心权限 + 团队普通管理员权限（含小组全部 + RegularMember） |
| 组织普通管理员(L6) | 组织普通权限 + 团队普通管理员权限（含小组全部 + RegularMember） |
| 组织核心管理员(L7) | 组织核心权限 + 组织普通管理员权限 + 团队核心权限 + 小组核心权限 + RegularMember 权限 |
| SuperAdmin(L8) | 全部 45 个权限点 |

### 2.4 构成说明示例（组织核心管理员）

`organization_core_admin`(L7) 的有效权限集合 =
- 自身组织核心权限（org.update / org.role.assign / org.role.downgrade / org.team.create / org.read / org.member.*）
- ∪ 组织普通权限（org.read / org.member.invite / org.member.remove / org.member.list）
- ∪ 团队核心权限（team.update / team.role.assign / team.role.downgrade / team.group.create / team.archive / team.read / team.member.*）
- ∪ 小组核心权限（group.update / group.delete / group.role.assign / group.role.downgrade / group.read / group.member.*）
- ∪ RegularMember 权限（account.* / auth.logout / auth.refresh）

> 说明：因矩阵中各低层角色已在其自身域持有读取类权限，固化后高层角色自然覆盖；上述示例仅示意构成，实际以 [模块 README - 权限点矩阵](./权限管理模块-V1.0.0.md#五权限点矩阵) 为准。注意纵向继承仅取同层级，故组织核心包含团队核心 / 小组核心，但不额外包含超出团队核心所涵盖的团队普通权限（团队普通已含于团队核心）。

### 2.5 变更与刷新

- 继承关系在初始化时固化；若后续调整角色权限矩阵，需重新执行初始化使映射重新预填充。
- 重新预填充后，需失效相关账号的权限缓存（见 [02-权限点校验 - 校验性能](./02-权限点校验-V1.0.0.md)），确保下次请求加载最新权限。

---

## 三、边界与异常处理

| 场景 | 处理方式 |
|------|----------|
| 角色权限矩阵调整后未重新预填充 | 权限校验仍以旧映射为准，需重新执行初始化修正 |
| 重新预填充后缓存未失效 | 强制失效相关权限缓存 |
| Public 角色越级访问受保护接口 | 因 Public 仅持 3 个公开权限点，受保护接口应被拒绝 |
