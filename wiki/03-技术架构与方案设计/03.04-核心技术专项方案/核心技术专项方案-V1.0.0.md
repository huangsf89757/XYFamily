## 一、多租户隔离实现

### 1.1 核心原则

**应用层隔离**（非 PG RLS），通过三层校验确保租户间完全隔离：

```
隔离 = Header 声明 scope + Membership 校验 + 查询强制带 org_id
```

### 1.2 三层校验

| 层 | 实现 | 位置 |
|----|------|------|
| 第 1 层：scope 声明 | `X-Organization-ID` / `X-Team-ID` / `X-Group-ID` Header | HTTP 请求 |
| 第 2 层：membership 校验 | MembershipValidator 中间件（含层级继承） | 中间件 |
| 第 3 层：查询隔离 | Repository 层所有查询强制带 `org_id` 条件 | Repository |

### 1.3 实现

```go
// internal/repository/query.go

// 所有组织相关查询必须通过此方法构建，禁止直接写 WHERE
func WithOrgScope(query *gorm.DB, orgID string) *gorm.DB {
    if orgID == "" {
        return query.Where("1=0") // 无 scope 声明 → 返回空
    }
    return query.Where("org_id = ? AND deleted_at IS NULL", orgID)
}

// 错误示例（禁止）
teams := db.Find(&teams) // ❌ 无 org 过滤

// 正确示例
teams := WithOrgScope(db, req.OrgID).Find(&teams) // ✅
```

### 1.4 安全约束

- **禁止绕过**：任何跳过中间件的路由必须显式声明"公开"并审查
- **代码审查**：所有 Repository 查询必须通过 `WithOrgScope`，CI 中增加静态检查
- **审计**：跨 org 访问尝试记录到 `operation_audit_logs`（`details` 含请求 org 与实际 org）

## 二、RBAC 权限引擎

### 2.1 架构

```
权限校验流程（p95 目标 < 100ms）：

1. 中间件从 context 获取 account + scope
2. 查询 Redis 缓存 perm:account:{id}:org:{org_id}
3. 缓存命中 → 校验权限点 code ∈ permSet → 返回
4. 缓存未命中 → 层级回溯查询 DB → 构建 permSet → 写入 Redis 缓存 → 校验 → 返回
```

### 2.2 层级回溯算法

```go
// internal/service/rbac/resolution.go

type Scope struct {
    OrgID   string
    TeamID  string
    GroupID string
}

type MemberInfo struct {
    Role        string // Public / RegularMember / group_ordinary_admin / group_core_admin / team_ordinary_admin / team_core_admin / organization_ordinary_admin / organization_core_admin / SuperAdmin
    ContainerID string
    Container   string // "org" / "team" / "group"
}

// Resolve 返回 account 在 scope 下的有效角色和权限点集合
func (s *RBACService) Resolve(accountID uuid.UUID, scope Scope) (*MemberInfo, map[string]bool, error) {
    // 1. SuperAdmin 直出
    if s.isSuperAdmin(accountID) {
        return &MemberInfo{Role: "SuperAdmin"}, allPermissions(), nil
    }

    // 2. 按 scope 层级向下回溯
    if scope.GroupID != "" {
        // a. 直接小组成员
        if m := s.findDirectMember(accountID, "group", scope.GroupID); m != nil {
            return m, s.buildPermSet(m.Role)
        }
        // b. 团队管理员（可访问下属所有小组）
        if scope.TeamID != "" {
            if m := s.findDirectMember(accountID, "team", scope.TeamID); m != nil && m.Role == "team_core_admin" {
                return m, s.buildPermSet(m.Role)
            }
        }
        // c. 组织管理员（可访问下属所有小组）
        if scope.OrgID != "" {
            if m := s.findDirectMember(accountID, "org", scope.OrgID); m != nil && m.Role == "organization_core_admin" {
                return m, s.buildPermSet(m.Role)
            }
        }
    }

    if scope.TeamID != "" {
        if m := s.findDirectMember(accountID, "team", scope.TeamID); m != nil {
            return m, s.buildPermSet(m.Role)
        }
        if scope.OrgID != "" {
            if m := s.findDirectMember(accountID, "org", scope.OrgID); m != nil && m.Role == "organization_core_admin" {
                return m, s.buildPermSet(m.Role)
            }
        }
    }

    if scope.OrgID != "" {
        if m := s.findDirectMember(accountID, "org", scope.OrgID); m != nil {
            return m, s.buildPermSet(m.Role)
        }
    }

    // 3. Public 角色兜底（无 membership 时）
    return &MemberInfo{Role: "Public"}, s.buildPermSet("Public"), nil
}
```

### 2.3 权限点构建

```go
// internal/service/rbac/permissions.go

func (s *RBACService) buildPermSet(roleCode string) map[string]bool {
    // 权限继承：L5 → L4 → L3 → L2 → L1 → L0
    levels := map[string]int{
        "SuperAdmin":                 8,
        "organization_core_admin":    7,
        "organization_ordinary_admin": 6,
        "team_core_admin":            5,
        "team_ordinary_admin":        4,
        "group_core_admin":           3,
        "group_ordinary_admin":       2,
        "RegularMember":              1,
        "Public":                     0,
    }
    level := levels[roleCode]

    // 从 DB 查询所有 role.level <= 当前 level 的权限点
    perms := s.repo.LoadPermissionsUpToLevel(level)

    result := make(map[string]bool)
    for _, p := range perms {
        result[p.Code] = true
    }
    return result
}
```

### 2.4 权限继承规则

| 角色 | level | 继承 |
|------|-------|------|
| Public | 0 | 仅自身（auth.register/login/reset_password） |
| RegularMember | 1 | L0 + L1（account 全部 + 基础 read） |
| group_ordinary_admin | 2 | L1 + L2（成员管理 + 资源读取） |
| group_core_admin | 3 | L2 + L3（+ group 全部，含结构性操作） |
| team_ordinary_admin | 4 | L3 + L4（成员管理 + 资源读取） |
| team_core_admin | 5 | L4 + L5（+ team 全部，含结构性操作） |
| organization_ordinary_admin | 6 | L5 + L6（成员管理 + 资源读取） |
| organization_core_admin | 7 | L6 + L7（+ org 全部，含审计查看） |
| SuperAdmin | 8 | 全部 45 个权限点 |

## 三、JWT 鉴权链

### 3.1 Token 生命周期

```
登录成功
  → 签发 Access Token (TTL=30min) + Refresh Token (TTL=7d, 存入 sessions)
  → 返回给客户端

Access Token 过期
  → 客户端用 Refresh Token 请求 /auth/refresh
  → 后端验证 Refresh Token（查 sessions 表）
  → 签发新 Access Token + 轮换 Refresh Token（Refresh Rotation）

登出 / 改密 / 强制下线
  → 写入 Redis 黑名单（jti，TTL=Access Token 剩余有效期）
  → 删除 sessions 中的 Refresh Token（登出时）
```

### 3.2 Refresh Token Rotation

```go
// internal/service/auth/refresh.go

func (s *AuthService) Refresh(refreshToken string) (*TokenPair, error) {
    // 1. 查询 sessions 表
    session, err := s.repo.FindSessionByRefreshToken(refreshToken)
    if err != nil {
        return nil, errors.New(101004, "Refresh Token 无效")
    }

    // 2. 校验：未撤销 + 未过期
    if session.Revoked {
        return nil, errors.New(101005, "Refresh Token 已撤销")
    }
    if time.Now().After(session.ExpiresAt) {
        s.repo.DeleteSession(session.ID)
        return nil, errors.New(101006, "Refresh Token 已过期")
    }

    // 3. 轮换：撤销旧 Refresh Token，签发新的
    s.repo.RevokeSession(session.ID)
    newPair := s.signNewToken(session.AccountID)
    s.repo.CreateSession(session.AccountID, newPair.RefreshToken, 7*24*time.Hour)

    return newPair, nil
}
```

**轮换意义**：每次刷新都更换 Refresh Token，旧 Token 立即失效。攻击者盗用 Refresh Token 后只能使用一次（第二次刷新时发现已撤销）。

### 3.3 Token 签发

```go
// pkg/jwt/sign.go

type Claims struct {
    AccountID string            `json:"account_id"`
    OrgIDs    []string          `json:"org_ids"`
    Roles     map[string]string `json:"roles"` // org_id → role
    JTI       string            `json:"jti"`
    jwt.RegisteredClaims
}

func SignToken(account *Account, memberships []*Membership) (*TokenPair, error) {
    jti := uuid.New().String()
    iat := time.Now()
    exp := iat.Add(30 * time.Minute)

    claims := &Claims{
        AccountID: account.AccountID.String(),
        OrgIDs:    extractOrgIDs(memberships),
        Roles:     extractRoles(memberships),
        JTI:       jti,
        RegisteredClaims: jwt.RegisteredClaims{
            IssuedAt: jwt.NewNumericDate(iat),
            ExpiresAt: jwt.NewNumericDate(exp),
            Issuer:   "xyfamily",
        },
    }

    accessToken, _ := jwt.NewWithClaims(jwt.SigningMethodHS256, claims).SignedString([]byte(secret))
    refreshToken := uuid.New().String() // 随机生成，有状态存储在 sessions

    return &TokenPair{
        AccessToken:  accessToken,
        RefreshToken: refreshToken,
        ExpiresIn:    1800, // 30min
    }, nil
}
```

## 四、审计日志实现

### 4.1 双表设计

| 表 | 记录内容 | 写入时机 |
|----|---------|---------|
| `login_audit_logs` | 登录/登出/刷新（含成功/失败/失败原因/IP/User-Agent） | 同步（操作快，直接写入） |
| `operation_audit_logs` | 业务操作（含操作类型/目标/详情 JSONB） | 异步（Redis Stream → Consumer → DB） |

### 4.2 登录审计（同步写入）

```go
// internal/service/auth/login.go

func (s *AuthService) loginResult(account *Account, method, result, reason, ip, ua string) {
    s.repo.InsertLoginAuditLog(&LoginAuditLog{
        AccountID:    account.ID,
        LoginMethod:  method,
        LoginResult:  result, // "success" / "failed" / "locked"
        FailureReason: reason,
        IPAddress:    ip,
        UserAgent:    ua,
        CreatedAt:    time.Now(),
    })
}
```

### 4.3 操作审计（异步写入）

见 [中间件专项方案 - 审计日志中间件](../03.03-中间件专项方案/中间件专项方案-V1.0.0.md)。

### 4.4 审计日志查询

```sql
-- 查询某组织近 7 天的操作日志
SELECT account_id, operation_type, target_type, target_id, details, ip_address, created_at
FROM operation_audit_logs
WHERE org_id = :org_id
  AND created_at > NOW() - INTERVAL '7 days'
ORDER BY created_at DESC
LIMIT 50 OFFSET 0;
```

### 4.5 数据保留

| 策略 | 规则 |
|------|------|
| 保留期 | 1 年 |
| 清理方式 | 定时任务按月归档，超期记录标记 `deleted_at`（软删除） |
| 匿名化 | 对应账号注销后，audit log 中 `account_id` 保留，账号 PII 已匿名化（审计上下文不丢失） |
